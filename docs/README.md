# PDF分割命名ツール ドキュメント索引

このフォルダでは、現行MVPの仕様と実装状況を整理する。過去の調査、復元計画、旧仕様、チェックリストは履歴資料として `archive/` に保存し、現行仕様の判断には使わない。

## 読む順番

1. `01_要件定義書.md`: 現行MVPの正本。実装・受入判断ではこの文書を優先する。
2. `05_実装状況.md`: 現在の実装済み範囲、未受入項目、検証コマンド、後続候補を確認する。
3. `2026-05-31_デザインシステム.md`: MVP画面設計とCSS実装で守るUI方針を確認する。
4. `02_実装計画書.md`: 実装・検証計画、担当境界、回帰確認観点を確認する。

## 正本

- `01_要件定義書.md`: 現行MVPの要件定義書。STEP2検索支援、STEP3追加項目命名、状態保存、受入条件を含む。

## 現役の補助文書

- `05_実装状況.md`: 実装済み機能、未受入/手動確認項目、検証コマンド、後続候補をまとめる。
- `02_実装計画書.md`: 要件定義書を実装・検証へ落とし込む補助文書。STEP1-4の担当境界、既存実装の検証観点、desktop JS 回帰確認、要確認事項を整理する。
- `2026-05-31_デザインシステム.md`: MVP画面設計とCSS実装で守るUI方針。

## 方針・検討中

- `03_技術方針書.md`: 技術スタックと構造を既存実装に縛られず再検討した方針書。将来構想・移行判断用であり、現行MVP仕様の正本ではない。
- `04_移行ロードマップ.md`: 技術方針を段階移行へ落とし込むロードマップ。将来構想・移行判断用であり、現行MVP仕様の正本ではない。

## 運用文書

- `../apps/desktop/RELEASE.md`: Tauri updater、GitHub Releases、`latest.json`、署名鍵、v0.1.2以降の配布手順。既存テストが参照するため `docs/` へ移動しない。

## 参考・履歴

- `archive/`: 初期調査、復元計画、旧アプリ解析、旧仕様、社内展開メモ、各種チェックリスト。履歴資料であり、現行仕様の判断には使わない。
- `superpowers/plans/`: 過去の実装計画メモ。履歴資料であり、現行仕様の判断には使わない。

## アセット

- `assets/`: 仕様書や説明資料で参照する画像などの管理対象アセット。

## 更新ルール

- 現行MVPの仕様変更は、まず `01_要件定義書.md` に反映する。
- 実装済み/未受入/検証コマンドの変更は `05_実装状況.md` に反映する。
- UIの見た目やコンポーネント規則は `2026-05-31_デザインシステム.md` に反映する。
- リリース手順は `apps/desktop/RELEASE.md` に反映し、索引だけを本書に置く。
- 履歴資料は内容を書き換えず、必要に応じて `archive/` 内で参照する。
