## 1. 背景と目的

現状は `run_sidecar` コマンドがリクエスト毎に `py -3.12 -m pdf_splitter_tool --sidecar-request -` を新規起動する。Python インタプリタ起動＋ `import fitz`（PyMuPDF）の固定費が毎回かかり、ページ送り1回あたり推定 0.5〜1.5 秒を浪費している。

本設計の初版では、アプリ起動後に Python プロセスを1本だけ常駐させ、stdin/stdout の JSON Lines でリクエストをループ処理した。#113 の追補では、重い処理による head-of-line blocking を避けるため、対話レーンとバルクレーンをそれぞれ lazy 起動する2プロセス構成へ更新した。各レーン内では引き続き1リクエストずつ直列処理する。

- 対話レーン: プレビュー、ページ本文、ハイライト、PDF情報、状態保存/復元など
- バルクレーン: `preflight`、`export`、`blank_candidates`、`index_candidates`、`search_text`

この分離により、出力や全ページ走査の実行中でもプレビュー操作を処理できる。同時 export は同じバルクレーンで直列化されるため、既存の二重実行防止と request ID の整合性検証は維持される。

**変更影響は Rust（lib.rs）と Python（__main__.py 周辺）に閉じる。** フロントエンドは従来どおり `invoke("run_sidecar", { request })` を呼ぶだけで、リクエスト/レスポンスの中身（`sidecar.ts` の型）も一切変わらない。

## 2. 決定サマリ

| 項目 | 決定 | 理由 |
|---|---|---|
| プロトコル | ID 付きエンベロープの JSON Lines（1行1メッセージ・UTF-8・コンパクト JSON） | 行区切りが最も単純。ID で要求と応答の対応ずれ（desync）を検知できる |
| 並行性 | 対話/バルクの2レーン。各レーン内は1リクエストのみ in-flight | 重い出力・全ページ走査からプレビューを分離しつつ、同種処理の直列性と既存プロトコルを維持する |
| 起動タイミング | lazy（最初のリクエスト時に spawn） | アプリ起動を遅らせない。起動直後の `state_load` 呼び出しが実質ウォームアップになる |
| 終了処理 | 一次: stdin EOF で Python 側が自然終了。二次: Tauri `RunEvent::Exit` で両レーンを kill | 親プロセス消滅＝パイプ閉鎖＝EOF なので孤児プロセス化しない。kill は保険 |
| クラッシュ/ハング | リクエスト単位タイムアウト → kill → 当該リクエストはエラー応答 → 次リクエストで再 spawn | 自動リトライはしない（ユーザーが再操作すればよい社内ツール） |
| fitz ドキュメントキャッシュ | **今回は導入しない** | §7 参照。Windows ファイルロックの実害が利得を上回る |
| ワンショットモード | `--sidecar-request` は現状のまま残す | 既存テスト・デバッグ用。削除する理由がない |
| 切替手段 | 環境変数 `PDF_ORGANIZER_SIDECAR_MODE`（`resident` 既定 / `oneshot` フォールバック） | 常駐化に問題が出た場合の即時退避路 |
| 新規依存 | Rust crate 追加なし・Python パッケージ追加なし | std の thread / mpsc / Mutex と tauri 同梱機能で足りる |

## 3. プロトコル仕様（Rust ⇔ Python のインターフェース契約）

この章が両担当の契約。**ここに書かれていない挙動に依存しないこと。**

### 3.1 起動コマンド

```
py -3.12 -m pdf_splitter_tool --sidecar-serve
```

- 作業ディレクトリ: 現行ワンショットと同じ `recovery/`（`find_recovery_dir` の解決結果）
- 環境変数: 現行と同じ `PYTHONIOENCODING=utf-8` / `PYTHONUTF8=1` を付与
- `PDF_ORGANIZER_PYTHON` による Python 実体の上書きは現行どおり有効

### 3.2 メッセージ形式（JSON Lines）

- 1行 = 1メッセージ。行区切りは `\n`（Python の `print` / Rust の `write` とも LF を明示）
- エンコーディング: UTF-8。JSON は `ensure_ascii=False` 相当でよい（UTF-8 パイプ前提）
- コンパクト JSON（`separators=(",", ":")`）。JSON 文字列内の改行は標準の `\n` エスケープで表現されるため、1メッセージが物理1行に収まることが保証される

**リクエスト（Rust → Python stdin）**:

```json
{"id":1,"request":{"command":"page_preview","pdf_path":"C:\\...\\a.pdf","page_no":3}}
```

- `id`: 64bit 非負整数。Rust 側がプロセス起動ごとに 1 から単調増加で採番
- `request`: 既存の sidecar リクエストオブジェクトをそのまま格納（`sidecar.ts` の `SidecarRequest` と同一。中身の仕様変更はしない）

**レスポンス（Python stdout → Rust）**:

```json
{"id":1,"response":{"ok":true,"command":"page_preview","...":"..."}}
```

- `id`: 対応するリクエストの id をそのまま返す
- `response`: 既存の `handle_request` の戻り値をそのまま格納（`SidecarResponse` と同一）
- アプリケーションエラー（不正な command、ファイル不在等）は従来どおり `response.ok: false` で表現する。エンベロープ層のエラーではない

**プロトコル層エラー（行が JSON として解釈できない等）**:

```json
{"id":null,"response":{"ok":false,"command":"","error":"...","error_type":"..."}}
```

- Python はパース不能な行を受けても**プロセスを終了せず**、`id: null` のエラー応答を返して次の行の待機に戻る
- Rust は受信した `id` が送信した `id` と一致しない場合（`null` 含む）を desync とみなし、プロセスを kill して当該リクエストにエラーを返す（§4.3）

### 3.3 ストリームの規律

- **stdout はプロトコル専用**。Python 側はレスポンス行以外を stdout に書いてはならない（`print` デバッグ禁止。診断は stderr へ）
- **stderr は診断ログ専用**。Rust はプロトコルとして解釈せず、読み捨てつつログ転記する（§5.4）。Python はクラッシュ時のトレースバックを stderr に出してよい
- レスポンス書き込み後は**必ず flush** する（Python: `sys.stdout.flush()`）
- Python は **stdin の EOF を検知したら正常終了（exit 0）** する。これが常駐プロセスの一次終了経路

### 3.4 順序保証

- Rust は各レーンへ同時に1リクエストだけ送る（レーン内直列化）。各 Python プロセスは「1行読む → 処理 → 1行書く」の逐次ループでよく、単一プロセス内の並行処理・順序入替を考慮する必要はない

## 4. ライフサイクル

### 4.1 起動（lazy）

1. `run_sidecar` 呼び出し時、対象レーンの常駐プロセスが未起動 or 死亡済みなら spawn する
2. spawn 失敗（Python 不在等）はそのリクエストのエラーとしてフロントへ返す（現行と同じエラー面）
3. ウォームアップ専用の事前リクエストは設けない。対話レーンはアプリ起動直後の `state_load`、バルクレーンは最初の重処理でそれぞれ初回固定費を支払う

### 4.2 終了

| 経路 | 仕組み |
|---|---|
| 一次（常に効く） | アプリプロセス終了 → パイプ閉鎖 → Python が stdin EOF を検知して exit 0 |
| 二次（明示 kill） | `RunEvent::Exit` で両レーンの active child handle を明示 kill。request Mutex とは独立しているため処理中でも停止できる |

二次経路の実装メモ: `.build(...)` + `.run(|app_handle, event| ...)` 形式を使い、各レーンは request slot と active child handle を別の Mutex で保持する。`RunEvent::Exit` は request slot を取得せず active child handle を直接 kill するため、長時間処理の完了やコマンド別タイムアウトを待たずに終了できる。

### 4.3 異常検知と再起動

| 異常 | 検知 | 対応 |
|---|---|---|
| ハング | リクエスト単位タイムアウト（対話系は既定30秒、バルク系は既定300秒） | 対象レーンを kill + wait → 当該リクエストにタイムアウトエラーを返す → 次の同レーンリクエストで再 spawn |
| リクエスト中のプロセス死 | stdout reader が EOF / 書き込みの broken pipe | 当該リクエストに stderr の内容（取れれば）を含むエラーを返す → 次リクエストで再 spawn |
| desync（id 不一致・パース不能応答） | 受信エンベロープの id 検証 | kill → エラー応答 → 次リクエストで再 spawn |
| アイドル中のプロセス死 | 検知しない（監視スレッドは置かない） | 次リクエストの spawn チェックで自然回復 |

- **自動リトライはしない**。失敗はフロントにエラーとして見せ、ユーザーの再操作（ページ送り等）で再 spawn される。spawn はリクエスト契機のみなので再起動ループは構造上発生しない
- 再 spawn 時は採番 id をリセットしてよい（プロセスごとに独立した id 空間）

## 5. Rust 側設計（Rust 担当）

### 5.1 状態の保持

```rust
struct SidecarState {
    interactive: Arc<SidecarLaneState>,
    bulk: Arc<SidecarLaneState>,
}

struct SidecarLaneState {
    request_slot: Mutex<Option<ResidentSidecar>>,
    active_child: Mutex<Option<Arc<Mutex<Child>>>>,
}

struct ResidentSidecar {
    child: std::process::Child,
    stdin: std::process::ChildStdin,
    stdout_rx: std::sync::mpsc::Receiver<std::io::Result<String>>, // reader スレッドから行を受信
    next_id: u64,
}
```

- `tauri::Builder.manage(SidecarState::new())` で登録し、コマンドから `tauri::State<SidecarState>` で取得
- コマンド種別から対象レーンを選び、その `request_slot` だけを `spawn_blocking` クロージャ内でロックする。request Mutex がレーン内の直列化を実現し、`active_child` は終了イベントから処理中プロセスを停止するためだけに使う

### 5.2 リクエスト処理フロー

`run_sidecar` のコマンドシグネチャは **State 引数の追加のみ**（フロントから見た呼び出しインターフェースは不変）:

```rust
#[tauri::command]
async fn run_sidecar(
    state: tauri::State<'_, SidecarState>,
    request: serde_json::Value,
) -> Result<serde_json::Value, String>
```

処理手順（`spawn_blocking` 内）:

1. `PDF_ORGANIZER_SIDECAR_MODE` が `oneshot` なら現行のワンショット実装へ分岐（現行コードは関数として残す）
2. コマンド種別から対話/バルクレーンを選択し、対象 Mutex をロック → `Option` が `None` または死亡済みなら spawn（§3.1 のコマンド・cwd・env）
3. `id` 採番 → エンベロープを stdin に1行書いて flush
4. `stdout_rx.recv_timeout(timeout)` で1行待つ
5. 受信行をパース → `id` 検証 → `response` を取り出して返す
6. タイムアウト / EOF / desync は §4.3 のとおり kill して `Option` を `None` に戻し、エラー文字列を返す

### 5.3 stdout reader スレッド

Windows の同期パイプ読み取りにはタイムアウトを直接かけられないため、spawn 時に専用スレッドを1本立てる:

- `BufReader::lines()` で stdout を読み続け、1行ごとに `mpsc::Sender` へ送る
- EOF / エラーで送信を終えてスレッド終了（チャネル切断が Rust 側への死亡通知になる）
- リクエスト側は `recv_timeout(timeout)` で「タイムアウト」「行受信」「チャネル切断（プロセス死）」の3状態を区別できる

### 5.4 stderr reader スレッド

- spawn 時にもう1本、stderr を行単位で読み捨てるスレッドを立てる（パイプ詰まりによるデッドロック防止が主目的）
- 読んだ行は直近 N 行（例: 50 行）をリングバッファに保持し、プロセス死亡時のエラーメッセージに含める。実装は `Arc<Mutex<VecDeque<String>>>` 程度でよい

### 5.5 既存コードの扱い

- `wait_for_sidecar_output` 等のワンショット実装は削除せず `run_sidecar_oneshot` として温存（`oneshot` モードと将来のデバッグで使用）
- `find_recovery_dir` / `sidecar_python_launcher_from` / `sidecar_timeout_from` は両モード共通で再利用
- 既存の `#[cfg(test)]` テストはそのまま生かす

## 6. Python 側設計（Python 担当）

### 6.1 サーブループ

`__main__.py` に `--sidecar-serve` フラグを追加し、ループ本体はテスト可能性のため**ストリームを引数に取る関数**として切り出す（配置は `__main__.py` 内 or `sidecar.py` のどちらでも可。推奨: `sidecar.py` に `serve(stdin, stdout)` を追加）:

```python
def serve(stdin: TextIO, stdout: TextIO) -> None:
    for line in stdin:                      # EOF でループ終了 → 正常終了
        line = line.strip()
        if not line:
            continue                        # 空行は無視
        request_id, payload = _parse_envelope(line)   # 失敗時 request_id=None
        if payload is None:
            response = _error_response("", <パース例外>)
        else:
            response = handle_request(payload)        # 既存関数・変更なし
        stdout.write(json.dumps({"id": request_id, "response": response},
                                ensure_ascii=False, separators=(",", ":")) + "\n")
        stdout.flush()
```

設計上のポイント:

- **`handle_request` は一切変更しない**。既存の例外捕捉（コマンド内例外 → `ok: false` 応答）がそのまま効く
- エンベロープのパース失敗・`handle_request` の想定外例外を含め、**ループは例外で死なない**。必ず `id`（取れなければ `null`）付きエラー応答を書いて次の行へ進む
- stdout への書き込みは上記1箇所のみ。**ループ内外を問わず `print` を使わない**（プロトコル汚染防止）
- 起動時に `sys.stdout.reconfigure(encoding="utf-8", newline="\n")` 相当を行い、Windows での CRLF 変換を避ける

### 6.2 fitz ドキュメントキャッシュは導入しない（評価結果）

open 済み `fitz.Document` をプロセス内に保持すれば連続ページ送りはさらに速くなるが、今回は**見送る**:

1. **Windows のファイルロック**: fitz が PDF を開いている間、他プロセス（エクスプローラ・スキャナアプリ）からの削除・上書きが失敗する。本ツールの利用フローは「スキャン → 取込 → 分割」で、取込中に元 PDF を差し替える運用が現実にあり得るため、開きっぱなしの実害が大きい
2. **更新検知の複雑さ**: キャッシュを持つなら mtime/サイズによる無効化判定が必須になり、判定漏れは「古いプレビューを表示する」という発見しにくいバグになる
3. **利得が小さい**: 固定費の主因は Python 起動＋`import fitz`（0.5〜1.5 秒）であり、常駐化だけで解消する。残る `fitz.open` 自体は社内ツールの PDF サイズなら数十 ms 程度
4. 各コマンド（`page_count` / `page_preview_data_url` / `split_pdf`）は現状すべて `with fitz.open(...)` で即クローズしており、常駐化してもファイルロックは増えない。この性質は維持する

将来、計測でなお遅い場合に限り「直近1ファイルのみ・mtime/サイズで無効化する LRU(1)」を別 issue として検討する。

### 6.3 ワンショットモードとの共存

- `--sidecar-request` の処理は現状のまま残す（先に評価される分岐順も維持）
- `--sidecar-serve` と `--sidecar-request` が同時指定された場合は `--sidecar-request` を優先（既存分岐が先にあるため自然にそうなる。仕様として明記）

## 9. テスト戦略

### 9.1 Python（担当: Python 実装者）

1. **ループ単体テスト（subprocess 不要・高速）**: `serve(stdin, stdout)` に `io.StringIO` を渡す

2. **統合テスト（subprocess）**: `py -3.12 -m pdf_splitter_tool --sidecar-serve` を起動し、stdin に複数の実リクエスト（`pdf_info` → `page_preview` ×2 等）を流して応答検証 → stdin close → exit code 0 を確認。既存 `test_sidecar.py` のフィクスチャ PDF を再利用

### 9.2 Rust（担当: Rust 実装者）

1. エンベロープ生成・応答パース・id 検証の純粋関数を切り出して単体テスト
2. 偽サイドカー（`py -3.12 -c "...echo loop..."`）を使った reader スレッド + `recv_timeout` の統合テスト（既存 `sidecar_wait_drains_large_stdout_while_process_runs` と同じ流儀）

### 9.3 結合（両担当そろってから）

- `npm run tauri dev` 実機で: 起動 → ページ送り連打（応答速度の体感確認）→ export → アプリ終了後にタスクマネージャで python プロセスが残っていないこと
- タスクマネージャから python を強制 kill → 次のページ送りがエラー表示 → さらに次の操作で自動復旧すること

両者の唯一の接点は **§3 のプロトコル契約**。§3 に変更が必要になった場合は、片方が勝手に変えず本文書を更新して双方合意の上で進めること。

統合順序の推奨: Python 側を先にマージ（`--sidecar-serve` が存在しても Rust が呼ばなければ無影響）→ Rust 側マージ → §9.3 の実機結合確認 → issues.md の ISS-014 をクローズ。

## 11. リスク

| リスク | 影響 | 緩和 |
|---|---|---|
| Python 側ライブラリが stdout に書く（プロトコル汚染） | desync → リクエスト失敗＋再起動 | id 検証で検知し自動復旧。PyMuPDF の警告は stderr 行きのため通常発生しない。発生時は stderr ログで原因特定 |
| 環境変数（タイムアウト等）の変更が再起動まで反映されない | 軽微（社内ツール・再起動で解決） | 制約として周知。タイムアウト値はリクエスト毎に読むため影響は spawn 系のみ |
| `RunEvent::Exit` が異常終了時に走らない | python プロセス残留の懸念 | 一次経路（stdin EOF）が親プロセス消滅時にも機能するため、残留は実質発生しない |
| 常駐プロセスのメモリ増加（fitz の内部キャッシュ等） | 長時間使用でのメモリ増 | ドキュメントを都度クローズする現方針を維持。問題が出たら「Nリクエストごとに自主再起動」を検討（今回は実装しない） |
| 初回リクエストだけ従来どおり遅い | 体感は現状と同等（初回のみ） | 許容。気になる場合は eager 起動を別途検討（今回は lazy） |
