## 1. 本計画の前提

要件定義書の4工程（PDF取込・分割・入力・出力）は、`apps/desktop/`（Tauri + Next.js）と `recovery/pdf_splitter_tool/`（Python sidecar）に **すでに一通り実装されている**。

したがって本計画の主作業は「ゼロからの新規実装」ではなく、次の3つである。

1. 既存実装が要件定義書（特に §7 画面別要件・§11 受入条件）を満たすかの **検証**。
2. 要件を満たし切らない箇所の **補完**。
3. 担当境界・仕様の曖昧点の **合意と確定**。

そのため各STEPのタスクは「既存実装あり・回帰確認対象」と「新規/補完」に区別して記載する。コードに存在が確認できない事項や、正式な要件判断が必要な事項は「要確認」と明示し、推測で確定扱いしない。

## 2. 担当割り当て

担当は2名で分担する。本書では役割名で表記し、STEP1・STEP2を「担当A」、STEP3・STEP4を「担当B」とする。

| STEP | 工程 | 担当 | 要件定義書 |
| --- | --- | --- | --- |
| STEP1 | PDF取込 | 担当A | §7.1 |
| STEP2 | 分割 | 担当A | §7.2 |
| STEP3 | 入力 | 担当B | §7.3 / §8 |
| STEP4 | 出力 | 担当B | §7.4 / §9 |

依存関係は直列。前工程の出力（セグメントデータ・入力値）が次工程の入力になる。

```text
STEP1(取込) → STEP2(分割) → STEP3(入力) → STEP4(出力)
   担当A         担当A         担当B         担当B
```

中核データは `apps/desktop/app/page.tsx` の `buildSegments()` が生成する `allSegments`。STEP2の分割結果（`splitPointsByPdf`）と STEP1の共通入力（`commonMetadata`）、STEP3の個別入力（`segmentMetadata`）を合成し、STEP3/STEP4 はこれを前提に動作する。

## 4. STEP別タスク

### STEP1 PDF取込（担当A）

要件: §7.1 / 受入条件 #1・#2・#13

主な関係箇所:

- `apps/desktop/app/page.tsx`: `choosePdfs` / `chooseOutputDir` / `removePdf` / `clearPdfSelection` / `resetOutputDir` / `updateCommonMetadata` / `saveState` / `loadState` / `renderImportWork`
- sidecar: `pdf_info` / `state_save` / `state_load` / `StateManager`（`recovery/pdf_splitter_tool/state.py`）
- 型: `apps/desktop/lib/sidecar.ts` `AppPersistedState`

既存実装あり・回帰確認対象:

- PDF複数選択と重複パス排除。
- 選択PDFの一覧表示・1件除外・全クリア。
- 出力フォルダ選択・リセット。
- 共通入力欄（箱No・バインダーNo）の入力と保持。
- 状態保存・復元（tmp→本体置換＋bak退避、破損時のbak復元は既存実装とテスト補強で検証済み）。
- 「次へ進む条件」= PDFが1件以上 かつ 出力フォルダ指定あり。

新規/補完候補:

- [現行MVP方針] 状態復元時の欠損PDF処理。sidecar は `missing_input_paths` を返し、desktop は欠損分を除外して復元する。PDF一覧は追加順を保持し、復元時も保存済み `input_paths` の順序を維持したうえで欠損PDFだけを除外する。
- [現行MVP方針] 復元時、共通値は初期値として扱い、個別 `segmentMetadata` の値があるセグメントでは個別値を優先する。

スコープ外（§5）:

- 元PDFファイル名変更、プリセット管理・Legacyプリセットは実装しない。

完了条件: 現場PDFを1本以上選択し、出力先を指定して STEP2 へ進める。状態保存→アプリ再起動→復元が成立する（受入 #1・#2・#13）。

### STEP2 分割（担当A）

要件: §7.2 / 受入条件 #3・#4・#5・#6

主な関係箇所:

- `page.tsx`: `loadPreview` / `movePage` / `addSplitBeforeCurrentPage` / `undoLastSplit` / `splitEveryPage` / `splitPointsFor` / `buildSegments` / `renderSplitWork` / `renderSplitList`
- sidecar: `page_preview` / `PdfProcessor.page_preview_data_url`
- セグメント構造: フロント `SegmentView`、sidecar `Segment`（`models.py`）

既存実装あり・回帰確認対象:

- ページプレビュー表示と前後ページ移動（1..ページ数でクランプ）。
- 現在ページの前で分割（現在ページが次セグメントの開始ページになる。§7.2と一致）。
- 1ページごとに分割。
- 最後の分割の取り消し。
- 先頭ページ前は分割不可ガード。
- ページ番号は UI で終始1始まり。0始まり変換は sidecar 境界で扱い、`test_pdf_service.py` と sidecar contract テストで不変条件を回帰固定する。

新規/補完候補:

- [現行MVP方針] 複数PDF時の分割導線。分割操作は現在PDF単位で行い、セグメント一覧・再採番・出力はPDF表示順を横断して扱う。
- [現行MVP方針] 大判PDFでの連続ページ移動時のプレビュー体感は、sidecar 側のプレビュー画像上限（最大辺）と desktop 側のセッション内プレビューキャッシュで制御する。時間ベースの性能SLA化は後続判断とする。

スコープ外（§5）:

- ページ回転、ページ除外、ページ順入れ替え、現在ページだけ抽出、白紙検出は実装しない。

完了条件: 現場PDFを分割操作し、意図したページ構成のセグメント一覧が得られる（受入 #3・#4・#5・#6）。

引き継ぎ: H-1 は現行MVP方針として `segment-state.ts` / `segmentMetadata` のキー整合を前提にし、担当B側の入力・出力導線で回帰確認する。

### STEP3 入力（担当B）

要件: §7.3 / §8 命名仕様 / 受入条件 #7・#8・#9

主な関係箇所:

- `page.tsx`: `updateMetadata` / `resequence` / `previewFilename` / `padMetadata` / `missingMetadata` / `renderInputWork` / `renderInputList`
- sidecar: `build_yoshida_filename` / `REQUIRED_METADATA`（`processor.py`）
- 参考: `resequence_segments` は `workflow.py` に定義があるが sidecar コマンドとしては公開されていない。再採番はUI側 `resequence()` で完結する。

既存実装あり・回帰確認対象:

- セグメント一覧・ページ範囲・入力状態（OK/未入力）の表示。
- 箱No・バインダーNo・連番の入力（セグメント単位で保持）。
- 連番の再採番。
- 出力名プレビュー（テンプレート `{box_no:0>2}_{binder_no:0>2}_{seq:0>3}.pdf` のゼロ埋め）。
- 必須項目 = 箱No・バインダーNo・連番（UI と sidecar で項目一致）。

新規/補完候補（本STEP最大の整合性リスク）:

- [検証済み] **出力名プレビューと実出力名の一致**。UI 側プレビュー計算は `apps/desktop/lib/filename-policy.ts` に分離し、sidecar 側の命名仕様と同じ「ゼロ埋め後にファイル名正規化」方針で回帰テストする。
- [検証済み] 連番再採番は全PDF横断の表示順で行う。PDF単位再採番は後続検討。
- [検証済み] 再分割時、同一セグメントキーは全metadataを保持する。新セグメントは旧セグメントの `box_no` / `binder_no` を引き継ぎ、`seq` は重複防止のため空にする。PDFパスに `#` が含まれる場合も `segment-state.ts` のキー整合テストで回帰固定する。

スコープ外（§5・混入注意・最重要）:

- **入力補助候補は実装しない**（§5で後回し明記）。サジェスト・候補ドロップダウン等は作らない。MVPは手入力で3項目を入力できることが受入条件。
- OCR選択テキスト転記・詳細機能タブも作らない。

完了条件: 全セグメントに命名情報が入力され、正しい形式のファイル名プレビューが表示される。プレビュー名が STEP4 の実出力名と一致する（受入 #7・#8・#9）。

### STEP4 出力（担当B）

要件: §7.4 / §9 状態保存 / 受入条件 #10・#11・#12

主な関係箇所:

- `page.tsx`: `runPreflight` / `runExport` / `requestSegments` / `renderOutputWork` / `renderOutputList` / `canExport`
- sidecar: `preflight` / `export` / `check_segment_outputs` / `unique_output_path` / `ensure_unique_path` / `split_pdf`

既存実装あり・回帰確認対象:

- 出力前チェック（出力先未指定・セグメントなし・必須未入力・`.pdf`終端・ページ範囲超過を検出）。
- 一括出力（出力前チェック失敗時は出力ブロック）。
- 出力ログ表示（作成/失敗件数、各セグメントのメッセージ）。
- 上書き防止の一意名（`_2`・`_3`）。バッチ内重複も予約して回避。
- Windowsファイル名正規化（sidecar 側で出力前に正規化、これが正本）。
- `canExport` = 出力前チェック実行済み かつ 要修正0件。

新規/補完候補（実害リスク）:

- [検証済み] **一意名採番は出力前チェック側へ集約**。`ExportPathPolicy` / `check_segment_outputs` が確定した出力パスを `export` が尊重する。`test_export_policy.py` / `test_output_name_parity.py` と追加テストで `_2`・`_3` 以降の採番を回帰固定する。
- [検証済み] 出力前チェックと出力操作が別リクエストになる通常導線では、出力時に再チェックして最新の一意名を確定する。出力直前に確定済みパスが既存化した場合は、別名へ逃げず失敗として返す（`test_export_edge_cases.py`）。
- 正規化後に名前が空になる場合の扱いは、現状「安全な既定名」採用で要件§8（出力不可または既定名）と整合。必須項目チェックで先に止まるため到達経路は限定的。

スコープ外（§5）:

- 出力履歴画面、既存ファイルの「再利用」「スキップ」選択は実装しない（自動で一意名付与のみ）。

完了条件: 未入力があれば出力前に止まり、全入力後はチェックを通過して一括出力でき、既存同名ファイルがあっても上書きせず一意名で出力される（受入 #10・#11・#12）。

## 8. 現行MVP方針と要確認事項

現行MVP方針:

1. PDF一覧は追加順を保持する。復元時も保存済み `input_paths` の順序を維持し、欠損PDFだけを除外する（`missing_input_paths` と desktop 側除外復元で検証済み）。
2. 連番再採番は全PDF横断の表示順で行う。PDF単位再採番は後続検討。
3. 再分割時、同一セグメントキーは全metadataを保持する。新セグメントは旧セグメントの `box_no` / `binder_no` を引き継ぎ、`seq` は重複防止のため空にする。
4. 出力名プレビューは desktop 側の `filename-policy` と sidecar 側の命名仕様を回帰テストで揃える。
5. 一意名採番は出力前チェック側で確定し、出力実体は確定済みパスを尊重する。`export_policy` / `output_name_parity` 系テストで検証済みとし、確定済みパスが出力直前に既存化した場合の失敗扱いも `test_export_edge_cases.py` で検証済み。
6. 共通値はセグメントの初期値であり、個別 `segmentMetadata` がある場合は個別値を優先する。
7. 複数PDF時の分割操作は現在PDF単位で行い、一覧・再採番・出力はPDF表示順を横断して扱う。
8. 状態復元時の欠損PDFは sidecar の `missing_input_paths` と desktop 側の除外復元で扱う。
9. `_pdf_split_state.json` の主要フィールドと `common_metadata` / `segment_metadata` 内部値は `state_schema.py` で検証し、壊れた値を復元結果へ混入させない。
10. 大判PDFのプレビューは sidecar 側のプレビュー画像上限（最大辺）と desktop 側のセッション内プレビューキャッシュ/リクエストゲートで制御する。MVPの固定範囲は、sidecar unit coverage、`handle_request` 経由の sidecar-flow smoke、desktop pure helper/unit regression coverage とする。時間ベースの性能SLA化は後続判断とする。実ブラウザ/Tauri上のプレビュー導線全体のE2Eスモーク確認は、必要に応じた後続検証とする。

要確認事項（仕様判断の委譲先）:

以下は要件定義書に明示がなく、実装で確定が必要。担当者間で合意のうえ要件定義書または本計画へ反映する。

1. 大判PDFプレビューの時間ベース性能SLAを正式要件化するか（MVP後判断）。MVPでは sidecar unit coverage、`handle_request` sidecar-flow smoke、desktop pure helper/unit regression coverage を自動テスト範囲とする。
