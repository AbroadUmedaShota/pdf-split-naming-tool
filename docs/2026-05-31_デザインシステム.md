# PDF分割命名ツール デザインシステム

作成日: 2026-05-31
位置づけ: 最小構成MVPの画面設計とCSS実装で守るデザイン方針。

## 1. 基本方針

本ツールは、PDFプレビュー、分割一覧、入力表、出力確認を扱うデスクトップ業務アプリである。見た目はモダンSaaS寄りにするが、装飾よりも作業効率、視認性、状態判別を優先する。

主対象は1024px以上のPC/ノートPC幅とする。スマホ相当幅は表示崩れ防止のみを確認し、実作業向けの専用最適化は行わない。

## 2. トークン

CSSの `:root` に以下のトークン群を持つ。

- Color: 背景、surface、文字、境界、アクセント、状態色。
- Space: `--space-1` から `--space-5`。
- Radius: `--radius-1` と `--radius-2`。角丸は8px以下を基本にする。
- Typography: UIフォント、caption、control、body、title、page heading。
- Elevation/Motion: パネル影、ボタン影、短いtransition。

色の役割:

- ティール: 主操作、現在位置、ブランドアクセント。
- ブルー: 選択中行、情報的な強調。
- グリーン: 完了、OK、成功。
- アンバー: 要確認、未入力、注意。
- レッド: エラー、出力不可。

## 3. コンポーネント規則

横ステッパー:

- 4ステップを上部に横並びで表示するが、現場作業中は薄い進捗タブとして扱う。
- 各ステップは `作業中`、`完了`、`要確認`、`未着手` の状態を持つ。
- 現在ステップはティール系の背景と境界で示す。

トップサマリー:

- PDF件数、分割数、入力状態、出力先はヘッダー内の小さな作業サマリーとして表示する。
- 未設定や未入力はアンバー系で注意喚起する。

作業台レイアウト:

- 左ペインは一覧専用とし、PDF一覧、分割セグメント、入力対象、出力予定をStepに応じて切り替える。
- 中央ペインは主作業専用とし、取込設定、PDFプレビュー、命名入力、出力確認を配置する。
- 右ペインは常設しない。進捗条件と次アクションは各Stepの作業面または下部フッターに置く。
- 保存、復元、更新確認、開発用切替は補助操作として主導線から外す。
- 主アクションは作業フッターで最も見つけやすいCTAとして扱い、補助ボタンより大きさ、色、文字の太さを強める。
- 左ペインの選択中行は背景色だけでなく、左端のアクセント線で現在対象を示す。
- 中央ペインは他ペインより少し明るく、影と境界を強めて主作業面として見えるようにする。

パネル:

- 主要作業単位を1つの白いsurfaceとしてまとめる。
- パネル内にさらにカードを入れ子にしすぎない。
- セクション見出し、主操作、本文の順で配置する。
- Step 1、Step 3、Step 4の中央ペインは内容量に応じた高さにし、PDF未選択時に大きな空白を作らない。
- 大きな高さを確保するのはStep 2のPDFプレビュー領域に限定する。

ボタン:

- 主操作はティールのprimary。
- 補助操作は白背景のsecondary。
- 無効状態はopacityで抑え、操作できないことを明確にする。
- アイコンは `lucide-react` を使い、16px前後で統一する。

入力欄:

- ラベルは小さく、入力値は十分に読みやすくする。
- focus時はティールのoutlineで現在位置を示す。

テーブル/一覧:

- 業務情報はカードグリッドではなく行単位で扱う。
- 選択中行は淡いブルーで示す。
- 状態列には `OK`、`未入力`、`出力可能`、エラー文言を表示する。

空状態:

- PDF未選択、分割なし、入力対象なし、出力前チェック待ちは専用の空状態として表示する。
- 空状態には、次に行う主要操作を1つだけ置く。

## 4. 画面別適用

PDF取込:

- PDF選択を最も目立つprimary操作にする。
- PDF一覧、出力先、共通項目を1画面内で確認できる密度にする。
- 保存/復元は補助操作として下部に置き、分割へ進む導線を優先する。
- PDF、出力先、共通項目の完了条件を主CTAの近くに表示する。
- 出力先リセットは安全操作として扱い、古い出力前チェック結果を残さない。

分割:

- PDFプレビュー領域を最も大きく扱う。
- 画面は左の分割対象/ページ状態一覧、中央のPDFプレビュー、右の分割設定/検索支援の3ペインを基本にする。
- 左ペインは複数ページの状態一覧として扱い、ページ番号、所属範囲、分割点、現在ページ、白紙候補を確認できるようにする。
- 中央ペイン上部にはページ番号入力、前後ページ移動、幅合わせ、全体表示、ズームを集約し、PDFプレビューの視認性を優先する。
- 右ペインは上から「現在ページ」「手動分割」「検索」「候補検索」「白紙候補」「OCRテキスト」「完了操作」の順に並べる。
- 検索は分割候補を探す補助機能として扱う。検索範囲は現在操作中PDFに固定し、ハイライト対象用語の選択、選択中用語チップ、結果一覧、候補検索、OCR本文表示は右ペインに置く。PDFプレビュー上の検索語ハイライトとOCR本文内強調を連動させる。
- 候補検索と白紙候補は自動分割しない。利用者が中央プレビューで確認してから、手動分割操作を行う前提で表示する。
- 誤操作復旧として、現在表示中PDFの分割を全解除できる操作を用意する。ただし主CTAの近くには置かず、通常の完了導線と視覚的に分離する。
- 入力へ進む導線は右ペイン下部の完了操作に固定し、危険操作と隣接させない。

入力:

- 選択中セグメントの出力名プレビューを上部に表示する。
- 左ペインのセグメント表にはページ範囲、箱No、バインダーNo、連番、状態を表示する。
- 中央ペインでは選択中セグメントの箱No、バインダーNo、連番を編集する。
- 未入力件数、入力完了件数、出力前チェックへ進む条件は作業フッターに表示する。

出力:

- 出力前チェックと出力実行を明確に分ける。
- 出力可能、要修正、既存ファイル有無を一覧で確認できるようにする。
- 左ペインに出力予定一覧、中央ペインにチェック結果と出力ログ、作業フッターに再チェックと出力実行を配置する。

## 5. 受入確認

- 1366px前後で、全画面の主要操作が見切れない。
- 1024px以上で、PDFプレビュー、一覧、入力表、出力確認が破綻しない。
- スマホ相当幅で、致命的な重なりやページ全体の横崩れがない。
- `npm run typecheck` と `npm run build` が通る。
- `git diff --check` が通る。
