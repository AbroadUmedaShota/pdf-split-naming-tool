## 1. 目的

PDFを取り込み、ページ単位で分割範囲を決め、箱No・バインダーNo・連番を基本に命名して、上書きせずにPDF出力する。必要に応じて、追加項目をファイル名の先頭または末尾へ挿入できる。

現段階では、元アプリの全機能再現よりも、ヨシダエルシス向けの基本運用を迷わず実行できることを優先する。

## 2. 対象利用者

- 社内で紙資料やスキャン済みPDFを案件・箱・バインダー単位に整理する担当者。
- PDFを確認しながら、分割位置と出力ファイル名を手作業で確定する担当者。
- 配布前チェック、問い合わせ対応、保守を行う開発・運用担当者。

## 3. 対象環境

- 主対象: Windows上で動作するデスクトップアプリ。
- 操作対象: 1024px以上のPC/ノートPC幅。
- 重点確認: 1366px前後の一般的な業務PC幅。
- 最低限確認: 小型ノートPC幅で主要操作が画面外へ破綻しないこと。
- 任意確認: スマホ相当幅で致命的な重なりや横方向のページ全体崩れがないこと。

スマートフォンでPDF分割・命名・出力の実作業を完結できることは、MVPの受入条件に含めない。

## 4. 実装構成

ユーザーが操作するアプリは `apps/desktop/` の Tauri + Next.js デスクトップアプリ1つに統一する。

`recovery/pdf_splitter_tool/` はPython sidecarとして扱う。PDFページ数取得、プレビュー生成、分割、命名、出力前チェック、状態保存を担当する処理部品であり、独立した2つ目のGUIアプリではない。

現行の主要sidecar機能:

| command | 役割 |
| --- | --- |
| `pdf_info` | PDFのページ数、ページ番号、命名テンプレートを返す。 |
| `page_preview` | 指定ページのプレビュー画像をData URLで返す。 |
| `page_thumbnail` | 指定ページのサムネイル画像をData URLで返す。 |
| `page_text` | 指定ページの既存テキスト層を返す。 |
| `search_text` | 事前OCR済みPDFまたはテキスト層ありPDFを対象に単語検索する。 |
| `search_highlights` | 検索語のページ内一致位置を矩形座標で返す。 |
| `index_candidates` | 固定キーワードに基づき、分割候補になりやすいページを返す。 |
| `blank_candidates` | 白紙候補ページを返す。候補は自動確定しない。 |
| `preflight` | 出力前チェックを行い、出力可否、出力予定パス、警告を返す。 |
| `export` | セグメントごとにPDFを分割出力する。 |
| `state_load` | 作業状態を読み込む。 |
| `state_save` | 作業状態を保存する。 |

## 5. MVP対象範囲

MVPで残す機能:

- PDFファイルを選択する。
- 出力フォルダを選択する。
- PDFページをプレビューする。
- 前ページ、次ページへ移動する。
- 現在ページの前で分割する。
- 最後の分割を取り消す。
- 分割セグメント一覧を表示する。
- ページ状態一覧から現在ページ、分割点、所属範囲を確認する。
- 事前OCR済みPDFまたはテキスト層ありPDFで、STEP2の単語検索、検索語ハイライト、OCR本文表示を使える。
- 固定キーワードによる候補検索で、分割候補ページを確認できる。
- 白紙候補を表示する。候補は利用者が確認してから手動で分割判断する。
- セグメントごとに箱No、バインダーNo、連番を入力する。
- 任意の追加項目を最大2件定義し、ファイル名の先頭または末尾へ挿入する。
- OCR本文から追加項目へテキストを転記する。
- 連番を再採番する。
- 出力ファイル名をプレビューする。
- 出力前チェックを行う。
- 一括出力する。
- 既存ファイルがある場合は `_2`, `_3` などで自動的に一意名を作る。
- `_pdf_split_state.json` に作業状態を保存する。

MVPで後回しにする機能:

- 画像PDFに対するOCRエンジン内蔵、Tesseract、外部OCR API連携。
- 固定3項目へのOCR選択テキスト転記。
- Step 3入力補助候補、サジェスト、候補ドロップダウン。
- 検索結果や白紙候補からの自動分割。
- ページ回転、ページ除外、ページ順入れ替え、現在ページだけ抽出。
- 元PDFファイル名変更。
- プリセット管理、Legacyプリセット。
- 出力履歴画面。
- 既存ファイルの「再利用」「スキップ」選択。
- 詳細機能タブ。
- 旧Tkinter GUIの復活。

後回し機能は、MVPが現場運用で成立することを確認してから、別Issueまたは別仕様として追加判断する。

## 6. 画面構成

画面は4工程に絞る。

1. PDF取込
2. 分割
3. 入力
4. 出力

画面レイアウトは横ステッパーとSTEP別作業台UIを基本とする。左固定ナビではなく、上部の横ステッパーで4工程を表示し、各工程で対象一覧、中央作業面、補助操作領域を使い分ける。

共通レイアウト:

- 上部: アプリ名、現在状態、4ステップの横ステッパー、PDF件数、分割件数、入力状態、出力先状態。
- 左ペイン: 各Stepの対象一覧。PDF一覧、分割セグメント、入力対象、出力予定を表示する。
- 中央ペイン: 各Stepの主操作。PDF選択、PDFプレビュー、命名入力、出力前チェックを表示する。
- 右ペイン: STEP2では分割設定、検索、候補、OCR本文、完了操作を表示する。STEP3では命名入力と追加項目を表示する。その他のSTEPでは完了条件、警告、次に行う主操作を表示する。

## 7. 画面別要件

### 7.1 PDF取込

目的:

- 処理対象PDFと出力先を指定する。

主な表示項目:

- PDF選択ボタン。
- 選択済みPDF一覧。
- 出力フォルダ選択ボタン。
- 共通入力欄: 箱No、バインダーNo。
- 状態保存ボタン。

主な操作:

- PDFを追加する。
- 選択PDFを一覧から外す。
- PDF一覧を全クリアする。
- 出力フォルダを指定する。
- 出力フォルダ指定をリセットする。
- 箱No、バインダーNoを入力する。
- 状態を保存する。

次へ進む条件:

- PDFが1件以上選択されている。
- 出力フォルダが指定されている。

### 7.2 分割

目的:

- PDFを見ながら分割位置を決める。

主な表示項目:

- PDFプレビュー。
- ページ状態一覧。
- 現在ページ番号。
- 前ページ、次ページボタン。
- ページ番号直接入力。
- ズーム、幅合わせ、全体表示。
- 現在ページの前で分割ボタン。
- 最後の分割を取り消すボタン。
- 選択分割を削除ボタン。
- 現在PDFの分割を全解除ボタン。
- 分割一覧。
- ハイライト対象用語の選択ボタン、選択中用語、検索結果一覧。
- 検索語ハイライト。
- 候補検索一覧。
- OCR本文表示。
- 白紙候補一覧。

主な操作:

- 前後ページへ移動する。
- 複数PDF時に前後PDFへ移動する。
- ページ番号を直接入力して移動する。
- プレビュー表示を幅合わせまたは全体表示へ切り替える。
- 現在ページの前に分割境界を追加する。
- 最後に追加した分割境界を取り消す。
- 選択した分割点を削除する。
- 現在PDFの分割を全解除する。
- 現在操作中PDFだけを対象に、選択中用語を検索する。
- 検索結果をクリックし、該当PDF・ページへ移動する。
- 現在ページ上の検索語一致箇所をハイライト表示する。
- インデックス、目次、表紙、区切り、No.、番号、会社名、書類名などの固定キーワードから分割候補を探す。
- 候補ページをクリックし、利用者がプレビュー確認後に手動で分割する。

ページ番号ルール:

- 画面表示は1始まり。
- 内部処理はPDFライブラリへ渡す直前に0始まりへ変換する。
- 「現在ページの前で分割」は、現在ページを次セグメントの開始ページにする。

検索支援の条件:

- 検索、検索語ハイライト、OCR本文表示、候補検索は、事前OCR済みPDFまたは既存テキスト層ありPDFを対象にする。
- 検索範囲は現在操作中PDFのみとし、全PDF横断検索は行わない。
- ハイライト対象用語は、契約書関連の組み込み候補と利用者が追加した任意用語から複数選択できる。
- 検索用語プリセットはPC単位の `localStorage` に保存し、キーは `pdf-organizer.step2SearchTerms.v1` とする。`_pdf_split_state.json` の状態保存仕様は変更しない。
- 本ツールは画像PDFへOCR処理を実行しない。テキスト層がない場合も、通常の分割、命名、出力作業は継続できる。
- 候補検索と白紙候補は自動分割しない。利用者がプレビューで確認し、必要な場合だけ手動で分割する。
- 検索結果選択時は、該当PDF・ページへ移動し、選択中検索語の一致箇所を現在ページ上に表示する。

### 7.3 入力

目的:

- 分割セグメントごとに命名に必要な情報を入力する。

主な表示項目:

- セグメント一覧。
- ページ範囲。
- 箱No。
- バインダーNo。
- 連番。
- 追加項目。最大2件。各項目はファイル名の先頭または末尾に挿入できる。
- 出力ファイル名プレビュー。
- 連番再採番ボタン。

主な操作:

- セグメントを選択する。
- 箱No、バインダーNo、連番を入力する。
- 追加項目を追加、削除、入力する。
- 追加項目の挿入位置を先頭または末尾から選ぶ。
- OCR本文の選択テキストを追加項目へ転記する。
- 連番を再採番する。
- 出力予定名を確認する。

必須項目:

- 箱No。
- バインダーNo。
- 連番。

追加項目は任意入力とし、空値は出力ファイル名から除外する。

### 7.4 出力

目的:

- 出力前チェックを行い、問題がなければ分割PDFを一括出力する。

主な表示項目:

- 出力前チェックボタン。
- 出力実行ボタン。
- 出力予定一覧。
- チェック結果。
- 出力ログ。

チェック内容:

- 出力フォルダが指定されている。
- 出力対象セグメントが存在する。
- 箱No、バインダーNo、連番が入力されている。
- 生成ファイル名が `.pdf` で終わる。
- Windowsファイル名として使えない文字が正規化される。
- 既存ファイルがある場合、上書きせず一意名で出力できる。

出力動作:

- セグメントごとにPDFを生成する。
- 既存ファイルは上書きしない。
- 同名ファイルがある場合は `_2`, `_3` などを付けて出力する。

## 8. 命名仕様

固定3項目の命名テンプレートは後方互換の正本として維持する。

```text
{box_no:0>2}_{binder_no:0>2}_{seq:0>3}.pdf
```

出力例:

```text
01_02_003.pdf
```

入力値は文字列として扱い、テンプレート適用時にゼロ埋めする。

Windowsファイル名として使えない文字は `_` に正規化する。正規化後に空になる場合は出力不可または安全な既定名として扱い、実装側の出力前チェック結果に従う。

追加項目:

- 追加項目定義は `affix_defs` として保持する。
- 最大2件まで定義できる。
- 各項目は `prefix` または `suffix` の位置を持つ。
- `prefix` は固定3項目の前、`suffix` は固定3項目の後ろへ `_` 区切りで挿入する。
- 同じ位置に複数項目がある場合は定義順に並べる。
- 追加項目が未定義または空値の場合は、従来の `01_02_003.pdf` 形式と同じ出力名になる。

## 9. 状態保存

保存対象:

- 入力PDF一覧。
- 出力フォルダ。
- 分割セグメント。
- セグメントごとの箱No、バインダーNo、連番。
- 追加項目定義。
- 現在PDF。
- 現在ページ。

保存ファイル:

- `_pdf_split_state.json`
- `_pdf_split_state.bak.json`
- `_pdf_split_state.tmp`

`_pdf_split_state.json` の主要フィールド:

- `input_paths`: 入力PDF一覧。追加順を保持する。
- `output_dir`: 出力フォルダ。
- `split_points_by_pdf`: PDFごとの分割位置。
- `common_metadata`: 共通の箱No、バインダーNo。
- `segment_metadata`: セグメントごとの箱No、バインダーNo、連番。
- `affix_defs`: 追加項目定義。最大2件。`key`、`label`、`position` を持つ。
- `current_pdf`: 現在PDF。
- `current_page`: 現在ページ。

保存方針:

- 一時ファイルへ書き出してから本体へ置き換える。
- 既存状態ファイルはバックアップへ退避する。
- 破損した状態ファイルは退避し、読み込み可能なバックアップまたは一時ファイルから復元する。
- 保存失敗時に作業状態を壊さないことを優先する。
- 保存・復元時は主要フィールドと `common_metadata` / `segment_metadata` 内部の値をスキーマ検証し、型不整合や壊れた命名値を復元結果へ混入させない。

## 10. 非機能要件

安全性:

- 既存の出力PDFを上書きしない。
- 出力前チェックで未入力、出力先未指定、出力対象なしを検出する。
- 状態保存は一時ファイルとバックアップを使い、保存失敗時の破損リスクを下げる。

操作性:

- 4工程の現在位置と完了状態を横ステッパーで把握できる。
- 左一覧、中央作業、右進捗の3ペインで、対象・操作・次アクションを分ける。
- 業務PC幅で主要操作が見切れないことを優先する。

更新/配布:

- デスクトップアプリは Tauri updater と GitHub Releases の `latest.json` を使って更新確認できる。
- リリース手順、署名鍵、manifest生成、公開後確認は `apps/desktop/RELEASE.md` に従う。
- リリースassetのURL確認だけでは、アプリ内updaterのE2E確認は完了扱いにしない。

保守性:

- ユーザー向けGUIは `apps/desktop/` に集約する。
- PDF処理はPython sidecarへ寄せ、UIと処理責務を分離する。
- 旧配布物解析や復元メモは正本要件から分離し、`docs/archive/` に保存する。

## 11. 受入条件

1. PDFを選択できる。
2. 出力フォルダを指定できる。
3. PDFページをプレビューできる。
4. 現在ページの前で分割できる。
5. ページ移動、分割取り消し、選択分割削除、現在PDFの分割全解除ができる。
6. 分割セグメント一覧を確認できる。
7. 箱No、バインダーNo、連番を入力できる。
8. 連番を再採番できる。
9. `01_02_003.pdf` 形式の出力名を作れる。追加項目がある場合は先頭または末尾へ挿入した出力名を作れる。
10. 未入力がある場合は出力前に止められる。
11. 既存ファイルを上書きせず、一意名で出力できる。
12. 一括出力で分割PDFを生成できる。
13. 作業状態を保存し、再起動後に復元できる。
14. 1024px以上のPC/ノートPC幅で主要操作が成立する。
15. スマホ相当幅では、実作業最適化ではなく、致命的な表示崩れがないことを確認する。
16. テキスト層ありPDFでSTEP2の検索結果を確認し、結果クリックで該当ページへ移動できる。
17. 検索語ハイライト、OCR本文内の検索語強調、候補検索が表示され、候補が自動分割されない。
18. Tauri updater のリリース手順を参照でき、旧バージョンからの更新検出、インストール、再起動後のバージョン表示を人間が確認できる。
