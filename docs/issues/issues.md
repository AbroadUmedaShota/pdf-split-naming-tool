## P1（今回修正対象）

### ISS-001: サイドカー処理の多重実行ガードがなく、出力二度押しで全ファイルが複製される

- **優先度**: P1（Critical）
- **観点**: UX / 速度 / 保存
- **元指摘**: PERF-4, UX-1, UI-13
- **場所**: `apps/desktop/app/page.tsx` runExport / runPreflight / 各ボタン
- **ステータス**: CLOSED（2026-06-12 修正・再レビュー承認）
- **概要**: invokeSidecar を伴う処理に in-flight 状態がなく、実行中もボタンが押せる。出力実行を二度押しすると `unique_output_path` が `_2` を採番して複製セットを「成功」として生成する。export 成功後も canExport が true のままで再出力できる。
- **再現手順**: 出力実行をクリック → 応答前に再クリック → `xx_2.pdf` 群が生成される。
- **修正方針**: busy 状態を導入し、サイドカー実行中は関連ボタンを無効化＋処理中表示。export 完了後は preflight 結果をクリアし再チェックを必須にする。
- **修正担当**: coder_frontend
- **クローズ条件**: 実行中の再クリックで二重実行されないこと。export 後に再チェックなしで出力できないこと。

### ISS-002: Tauri コマンドが同期実装でメインスレッドを最大30秒ブロックする

- **優先度**: P1（Critical）
- **観点**: 速度
- **元指摘**: PERF-1, PERF-8
- **場所**: `apps/desktop/src-tauri/src/lib.rs:12-51, 114-146`
- **ステータス**: CLOSED（2026-06-12 修正・再レビュー承認）
- **概要**: `run_sidecar` が非 async コマンドのためメインスレッドで実行され、サイドカー完了（通常数百ms〜、最悪30秒タイムアウト）までイベントループが停止する。20ms ポーリングもメインスレッド上で回る。
- **修正方針**: async コマンド化し、ブロッキング処理を `tauri::async_runtime::spawn_blocking` へ。ポーリングは `child.wait()` のブロッキング待ちに置換（リーダースレッドが別途あるためデッドロックしない）。
- **修正担当**: coder_api
- **クローズ条件**: `cargo test` パス。コマンド実行中に他の IPC が詰まらない構造になっていること。

### ISS-003: 分割点の変更で入力済みメタデータが消え、分割を戻すと古い値が復活する

- **優先度**: P1（High）
- **観点**: 保存 / UX
- **元指摘**: SAVE-2, SAVE-3, UX-2
- **場所**: `apps/desktop/app/page.tsx` segmentKey / buildSegments / 分割操作
- **ステータス**: CLOSED（2026-06-12 修正・再レビュー承認）
- **概要**: segmentMetadata のキーが `パス#開始-終了` のため、分割点変更で影響範囲のキーが変わり入力済み seq 等が UI から消える。古いキーのデータは残留するため、分割を取り消すと stale 値が無言で復活し、重複 seq でも `_2` 採番で出力まで到達する。orphan データは state ファイルにも永続化される。
- **再現手順**: セグメント 1-10 に seq 入力 → 分割追加（消失）→ 分割取り消し（復活）。
- **修正方針**: ①分割点変更時に当該PDFの現行キー集合外の segmentMetadata を削除（stale 復活防止）。②入力済みセグメントに影響する分割操作時は確認ダイアログを出す。③保存時に現行キーでフィルタ。
- **修正担当**: coder_frontend
- **クローズ条件**: 分割の追加→取り消しで古い値が復活しないこと。入力済みデータに影響する操作で警告が出ること。

### ISS-004: 状態保存・復元に例外処理がなく、失敗時に UI が無反応になる

- **優先度**: P1（High）
- **観点**: 保存
- **元指摘**: SAVE-1
- **場所**: `apps/desktop/app/page.tsx:643-684` saveState / loadState
- **ステータス**: CLOSED（2026-06-12 修正・再レビュー承認）
- **概要**: saveState は try/catch なし、loadState は最初の invokeSidecar が try の外。invoke が reject（Python 起動失敗・タイムアウト・JSON parse 失敗）すると unhandled rejection となり、ステータス表示が更新されない。「保存したつもりで未保存」事故に直結する。
- **修正方針**: 両関数を try/catch で包み、他ハンドラと同形式のエラーステータス表示に揃える。
- **修正担当**: coder_frontend
- **クローズ条件**: invoke 失敗時にエラーステータスが表示されること。

### ISS-005: 状態復元が all-or-nothing で、PDF 1件の欠損で全復元が失敗する

- **優先度**: P1（High）
- **観点**: 保存 / UX
- **元指摘**: SAVE-4, SAVE-8
- **場所**: `apps/desktop/app/page.tsx:658-684` loadState
- **ステータス**: CLOSED（2026-06-12 修正・再レビュー承認）
- **概要**: `Promise.all` のため保存済み PDF が1件でも削除・移動されていると復元が丸ごと失敗し、原因ファイルも表示されない。また保存後に PDF のページ数が減っていた場合、current_page が範囲外となり「復元エラー」表示になる（データ自体は復元済み）。
- **修正方針**: `Promise.allSettled` で取得できた PDF のみ復元し、欠損ファイルをステータスで列挙。current_page は復元先 PDF のページ数でクランプ。
- **修正担当**: coder_frontend
- **クローズ条件**: 一部欠損時に残りが復元され、欠損ファイルが報告されること。

### ISS-006: ページプレビューの連打が効かず、古い応答が新しい表示を上書きしうる

- **優先度**: P1（High）
- **観点**: 速度 / UX
- **元指摘**: PERF-3, UX-8
- **場所**: `apps/desktop/app/page.tsx` movePage / loadPreview / selectPdf
- **ステータス**: CLOSED（2026-06-12 修正・再レビュー承認）
- **概要**: movePage が応答到着後の currentPage 更新に依存するため、応答待ち中の連打が全て同じページを要求し1ページしか進まない。世代管理がなく、遅延した古い応答が新しいプレビューを上書きしうる（ISS-002 の async 化後に顕在化）。ローディング表示もない。
- **修正方針**: ①リクエスト世代カウンタ（useRef）で最新応答のみ反映。②currentPage は関数型更新で楽観的に先行更新。③プレビュー読込中の表示。④`パス#ページ` キーのプレビューキャッシュ（LRU 上限あり）で戻り操作を即時化。
- **修正担当**: coder_frontend
- **クローズ条件**: 連打した回数分ページが進むこと。古い応答で表示が巻き戻らないこと。

### ISS-007: ファイル名プレビューと実出力ファイル名が不一致になるケースがある

- **優先度**: P1（High）
- **観点**: UI / 保存
- **元指摘**: UI-1
- **場所**: `apps/desktop/app/page.tsx:88-108` / `recovery/pdf_splitter_tool/processor.py:37-51`
- **ステータス**: CLOSED（2026-06-12 修正・再レビュー承認）
- **概要**: フロントは trim 後に 0 埋め、Python は trim せず format 整形する。`" 1"` のような空白混入時、プレビューは `01_...pdf`、実出力は `1 _...pdf` となり、画面で見た名前と違うファイルが生成される。
- **修正方針**: Python 側 `build_yoshida_filename` で値を strip してから整形（正は Python）。フロント側も入力値の前後空白を統一的に扱う。両者の一致をテストで担保。
- **修正担当**: coder_backend（Python）+ coder_frontend（入力正規化）
- **クローズ条件**: 空白混入入力でプレビューと実出力名が一致すること。Python テストで検証済みであること。

### ISS-008: 出力の部分失敗が「完了」バッジ・無色ステータスで隠れる

- **優先度**: P1（High）
- **観点**: UI / UX
- **元指摘**: UI-2, UI-3, UX-4
- **場所**: `apps/desktop/app/page.tsx:382-384, 637, 686-692`
- **ステータス**: CLOSED（2026-06-12 修正・再レビュー承認）
- **概要**: ①stepState が `created` の truthy のみで「完了」判定し、5件中2件失敗でも緑バッジになる。判定ブロックに `stepId === "output"` ガードがない。②ステータス色を文字列マッチ（`includes("エラー")` 等）で決めており、出力失敗時の「出力結果を確認してください。」が無色表示。ファイル名に「エラー」を含むと誤って赤になる。
- **修正方針**: ①失敗 0 かつ作成 >0 のみ done、失敗ありは attention に。②setStatus をトーン明示引数方式に変更し文字列推測を廃止。③失敗時は失敗ファイルを結果表示に列挙。
- **修正担当**: coder_frontend
- **クローズ条件**: 部分失敗時に attention 表示・警告トーンになること。

### ISS-011: state 保存の復旧順序の穴と多重起動時の tmp 書き込みレース

- **優先度**: P2（Medium）
- **観点**: 保存
- **元指摘**: SAVE-5, SAVE-6
- **場所**: `recovery/pdf_splitter_tool/state.py`
- **ステータス**: CLOSED（2026-06-12 修正・再レビュー承認）
- **概要**: ①save の「state→bak ローテート後、tmp→state コミット前」にクラッシュすると、load が bak（1世代前）を優先し最新の tmp に到達しない。残骸 tmp が後日古い状態として昇格する二次リスクもある。②tmp ファイル名が固定のため、アプリ2重起動で write-write レースが起き、別インスタンスの状態を「保存しました」と誤報告しうる。
- **修正方針**: ①load の優先順位を見直し、bak 採用前に検証済み tmp があれば tmp を優先（または更新時刻比較）。②tmp 名にプロセスIDサフィックスを付与。クラッシュ窓・多重起動のテストを追加。
- **修正担当**: coder_backend
- **クローズ条件**: 追加テスト含む recovery/tests/ が全パス。クラッシュ窓シナリオで最新保存が復元されること。

### ISS-012: CSP が null・process 権限が広い

- **優先度**: P2（Medium）
- **観点**: セキュリティ
- **元指摘**: SEC-1, SEC-5
- **場所**: `apps/desktop/src-tauri/tauri.conf.json` / `capabilities/default.json`
- **ステータス**: CLOSED（2026-06-12 修正・再レビュー承認）
- **概要**: CSP 無効のため、万一 XSS が成立すると `run_sidecar` 経由の任意パス書き込み・任意PDF読み取りへ連鎖する。process 権限は relaunch しか使わないのに default 一式が許可されている。
- **修正方針**: CSP に最小ポリシーを設定（`img-src` に `data:` が必要。Tauri はビルド時に script ハッシュを自動注入）。process 権限を relaunch 相当に縮小。ビルドが通ることを確認し、実機での画面表示・更新機能の動作確認は要実施項目としてイシューに記録。
- **修正担当**: coder_api
- **クローズ条件**: ビルド成功＋設定が最小権限になっていること。実機確認は人間の確認項目として残す。

### ISS-013: プレビュー生成の無駄（PDF 二重 open・PNG 固定・非コンパクト JSON）

- **優先度**: P2（Medium）
- **観点**: 速度
- **元指摘**: PERF-6, PERF-7
- **場所**: `recovery/pdf_splitter_tool/sidecar.py:58-71` / `processor.py:75-82` / `__main__.py:41`
- **ステータス**: CLOSED（2026-06-12 修正・再レビュー承認）
- **概要**: page_preview がバリデーション用と描画用で同じ PDF を2回 open する。プレビューは PNG 固定でスキャン画像だと数百KB〜数MB になり、base64+JSON で多段コピーされる。応答 JSON は indent=2 で、未使用の page_numbers 配列が大ページ数 PDF で肥大化する。
- **修正方針**: ①open を1回に統合。②プレビューを JPEG（quality 75 前後）化。③サイドカー応答をコンパクト JSON 化。④未使用 page_numbers の削除はフロント型と合わせて実施。
- **修正担当**: coder_backend
- **クローズ条件**: 既存テストパス＋プレビューが従来どおり表示されること。

### ISS-022: 部分復元後に保存すると欠損PDFの入力値が恒久消失する

- **優先度**: P3（発生条件が狭い） / **観点**: 保存 / **発見**: ISS-005 再レビュー（REG-1） / **ステータス**: CLOSED（2026-06-12 サイクル2で修正・再レビュー承認）
- 1件欠損で部分復元した後に「状態を保存」すると、欠損PDFのパスと入力済みメタデータが保存ファイルから消える（`split_points_by_pdf` だけ残り不整合）。一時的なドライブ切断で復活しうるファイルのデータが警告なしで失われる。
- 最小対応: 部分復元時の警告に「このまま保存すると読み込めなかったPDFの入力値は失われます」を追記。恒久対応: 欠損分の退避保存と `split_points_by_pdf` のフィルタ整合。

### ISS-023: クラッシュ残骸の pid 付き tmp が通常運転で掃除されない

- **優先度**: P3 / **観点**: 保存 / **発見**: ISS-011 再レビュー（REG-2） / **ステータス**: CLOSED（2026-06-12 サイクル2で修正・再レビュー承認）
- save() は自プロセスの tmp のみ扱い、state が正常なら load() も tmp に触れないため、クラッシュした別プロセスの tmp が work_dir に蓄積する。bak 不在時に古い残骸が最終手段として昇格しうる。
- 対応: save() の正常コミット後、コミット済み state より古い他 tmp を best-effort 削除（数行）。「bak より古い残骸 tmp が昇格しない」直接テストの追加も推奨。

### ISS-024: サイドカー応答待ち中の編集で古いチェック結果が適用されるレース

- **優先度**: P2 / **観点**: UX / 整合性 / **発見**: ISS-001 再レビュー（NOTE-1。修正前から存在する既存問題） / **ステータス**: CLOSED（2026-06-12 サイクル2で修正・再レビュー承認）
- preflight/export 応答待ち中も分割ボタン・メタデータ入力欄・再採番・出力フォルダ変更が操作可能で、編集後に届いた応答が編集前セグメントのチェック結果を `setPreflightChecks` で再セットする。結果、未チェックの状態で `canExport=true` になり「変更後は再チェック必須」の不変条件が破れる。
- 対応案: ①preflight にも世代カウンタ（送信時セグメントのスナップショット比較で不一致なら応答破棄）、②busy 中は編集系コントロールと出力フォルダ/リセットボタンも disabled（ノエル指摘の isBusy 漏れ2ボタン含む）。

### ISS-025: サイドカーのエラー応答（summary/checks 欠落）が型ガードを素通りしクラッシュしうる

- **優先度**: P2 / **観点**: UI / 堅牢性 / **発見**: ISS-008 再レビュー（既存の穴） / **ステータス**: CLOSED（2026-06-12 サイクル2で修正・再レビュー承認）
- runPreflight/runExport のガードが `response.command !== "..."` のみで、Python の `_error_response`（`{ok:false, command:"export", error:...}`、summary/items なし）を素通しする。export 中の例外（ソースPDF削除・ロック等）で `result.summary.failed` 参照が TypeError になり、不正な exportResult でレンダリングがクラッシュしうる。
- 対応: `response.ok` と `"summary" in response` / `"checks" in response` のガード追加。

### ISS-026: 再レビュー時の軽微な申し送り（まとめ）

- **優先度**: P3 / **観点**: UI / 速度 / **ステータス**: CLOSED（2026-06-12 サイクル2で修正・再レビュー承認）
- ステータスピルの warning トーン時もアイコンが中立（ClipboardCheck）のまま。AlertTriangle を warning にも使う。
- 成功系メッセージのトーン指定基準の揺れ（「出力フォルダを設定しました」= ok、「一覧から外しました」= 無指定）の統一。
- プレビューキャッシュの書込キーを応答側パスでなく要求側パスに統一（将来のパス区切り差異への備え）。
- `pageRequestRef`（楽観更新用）と `currentPage`（描画用 state）の並存理由を宣言行コメントで補足。

### ISS-027: リリースビルドでの環境変数オーバーライド無効化の検討

- **優先度**: P3 / **観点**: セキュリティ / **元指摘**: SEC-7（ISS-019 から分離） / **ステータス**: OPEN
- `PDF_ORGANIZER_PYTHON` で任意実行ファイルを Python ランチャとして起動できる。同一ユーザー権限内のため権限昇格にはならないが、配布版では `#[cfg(debug_assertions)]` でオーバーライドをデバッグビルド限定にする選択肢がある。導入時は `PDF_ORGANIZER_SIDECAR_MODE` 等の運用フォールバックとの兼ね合いを判断すること。

### ISS-028: サイクル2再レビューの軽微な申し送り（まとめ）

- **優先度**: P3 / **観点**: UI / 保存 / 速度 / **ステータス**: OPEN
- 即時対応済み（2026-06-12・本イシューの対象外）: PDF入替時の splitVisited リセット（ノエル NOTE-1）／resequence の busy ガード（NOTE-2）／serve ループの応答書き込みを try 内へ・BrokenPipe で静かに終了（スバル NOTE-P1）。
- 残項目:
  - preflight 送信シグネチャの組立式が3箇所に重複。共通ヘルパー化（ノエル NOTE-3。現状は fail-safe 方向のため実害なし）
  - 個別欄に空白のみを入れた場合、placeholder（共通値ヒント）が消えるが出力には共通値が使われる見た目のズレ（NOTE-4）
  - warning ボーダー色 `rgba(161,92,7,0.3)` の直書き3箇所のトークン化（NOTE-5）
  - sidecarError フォールバック文言が英語のまま（NOTE-6・到達性は実質ゼロ）

  - アイドル死で再 spawn する際に旧プロセスの stderr テールを破棄している（NOTE-R2）
  - load 時の tmp 検証が他プロセス書き込み中の tmp を .corrupt 退避しうる極小レース（NOTE-S1・ISS-011 承認済み構造の継承）
  - `--smoke` の state_tmp_path が `<pid>` 入りパターン文字列になった点の周知（NOTE-P2）

## 実機確認チェックリスト（コード検証では確定できない項目）

CSP・権限・UI レイアウトはコード上の検証を完了したが、以下は実機（`npm run tauri dev` / 本番ビルド）での確認が必要。

3. PDF ページプレビュー（data: URL 画像）の表示
4. 更新フロー一気通し: 更新確認 → インストール → relaunch 再起動（process:allow-restart の実機検証）
5. ファイル/フォルダ選択・確認ダイアログ（plugin-dialog open/confirm）の表示
6. 出力実行・ページ送り連打時の busy ガードと世代カウンタの体感確認
