## 1. 全体状況

ユーザー向けアプリは `apps/desktop/` の Tauri + Next.js デスクトップアプリに集約済み。PDF処理は `recovery/pdf_splitter_tool/` の Python sidecar が担当する。

sidecar の起動方式は v0.1.4 で常駐プロセス型（`--sidecar-serve`）へ移行し、v0.1.5 では対話処理とバルク処理を独立した2レーンへ分離した。各レーンは最初のリクエスト時に Python プロセスを lazy spawn し、stdin/stdout の JSON Lines でリクエストをループ処理する。これにより、出力や全ページ走査の実行中でもプレビューなどの対話処理を待たせない。詳細設計は `docs/2026-06-12_常駐サイドカー設計.md` を参照。フロントエンドからの呼び出し方（`invoke("run_sidecar", { request })`）と型（`sidecar.ts`）は変更なし。フォールバックとして `PDF_ORGANIZER_SIDECAR_MODE=oneshot` を設定すると旧ワンショット起動へ切り替えられる。

## 3. 主要仕様の実装状態

| 仕様 | 現在の扱い |
| --- | --- |
| STEP2検索範囲 | 現在操作中PDFのみ。旧UIの `現在PDF / 全PDF` 切替は使わない。 |
| 検索用語プリセット | ブラウザ `localStorage` の `pdf-organizer.step2SearchTerms.v1` に保存する。`_pdf_split_state.json` とは分離する。 |
| STEP2 sidecar | `page_thumbnail`、`page_text`、`search_text`、`search_highlights`、`index_candidates`、`blank_candidates` を追加済み。 |
| sidecar起動方式 | v0.1.4 で常駐プロセス型（`--sidecar-serve`）へ移行し、v0.1.5 で対話レーン（プレビュー等）とバルクレーン（出力・全ページ走査等）へ分離。各レーンを最初のリクエスト時に lazy spawn し、JSON Lines プロトコルでリクエストをループ処理する。ハング時は対象レーンだけをコマンド別タイムアウトで kill → 次リクエストで再 spawn し、アプリ終了時は処理中を含む両レーンを終了する。フォールバックは `PDF_ORGANIZER_SIDECAR_MODE=oneshot`。詳細は `docs/2026-06-12_常駐サイドカー設計.md`。 |
| sidecar配布 | `apps/desktop/scripts/build-sidecar.ps1` で `resources/sidecar/pdf-splitter-sidecar.exe` を生成し、Tauri bundle resources として同梱する。インストール版は同梱exeを優先し、開発時のみ `recovery/` + Python へフォールバックする。v0.1.4インストーラで同梱exeの `--smoke`、`pdf_info`、`page_preview`、インストール版UIからのSTEP1取込を確認済み。 |
| OCR方針 | 事前OCR済みPDFまたは既存テキスト層ありPDFの検索支援は実装済み。画像PDFへのOCRエンジン内蔵は後回し。 |
| 追加項目 | `affix_defs` 最大2件。位置は `prefix` または `suffix`。空値は出力名から除外する。 |
| 命名互換 | 追加項目なしでは `{box_no:0>2}_{binder_no:0>2}_{seq:0>3}.pdf` を維持する。 |
| 状態保存 | `_pdf_split_state.json` に `affix_defs` を含める。検索用語プリセットは含めない。 |
| リリース | `apps/desktop/RELEASE.md` に Tauri updater、署名鍵、GitHub Releases、`latest.json` の手順を集約する。 |

## 4. 検証コマンド

リポジトリ単位の自動回帰検証:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts\verify.ps1
```

desktop 側の個別確認:

```powershell
cd apps\desktop
npm run typecheck
npm run test:desktop
npm run test:step-render
npm run test:e2e
npm run test:installed-step1
npm run test:installed-core
npm run test:installed-output-recovery
npm run test:installed-state
npm run test:installed-native-dialog
npm run test:installed-updater
npm run release:manifest
npm run release:check
```

sidecar/API境界の重点確認:

```powershell
cd recovery
python -m pytest tests/test_sidecar.py tests/test_desktop_shell_contract.py
```

サンプルPDFのsidecarスモーク:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts\sample-pdf-smoke.ps1 -PdfPath "C:\path\to\sample.pdf"
```

注意:

- `scripts/verify.ps1` は Python pytest、desktop 側 JS 回帰テスト、desktop typecheck をまとめて確認する。
- `npm run release:manifest` は公開用 `release-assets` を現行バージョンの installer、`.sig`、`latest.json` の3ファイルだけで再生成する。
- `npm run release:check` は公開用 `release-assets` に旧版assetが混在していないこと、`latest.json` の version / URL / signature が現行 installer と `.sig` に一致することを確認する。
- `npm run test:installed-step1` はインストール済み v0.1.4 アプリを起動し、パスワード付きPDFの明示エラー、PDF以外/存在しないパスの明示エラー、エラー後の複数PDF再取込（一部失敗時に正常PDFだけ取込）、実PDFのSTEP1取込、複数PDFの連続追加、重複排除、追加順、現在PDFの選択状態維持、取込後の1件除外、全クリア、再取込、同梱sidecar利用を確認する。OSファイル選択ダイアログはテストハーネスで固定する。
- `npm run test:installed-core` はインストール済み v0.1.4 アプリを起動し、生成2ページPDFのSTEP1取込、STEP2分割、STEP3命名入力、STEP4出力前チェック、2件出力、出力後の `作成済み` 表示、出力実行ボタン無効化まで確認する。`PDF_ORGANIZER_INSTALLED_CORE_PDF_PATH` を指定すると、任意の実PDFで同じUI通し確認を実行できる。`PDF_ORGANIZER_INSTALLED_CORE_EXPECTED_PAGES` を指定した場合はページ数も厳密確認する。`PDF_ORGANIZER_INSTALLED_CORE_PREEXISTING_OUTPUT=1` を指定すると、出力先に同名PDFを事前配置して上書き防止を確認する。OSファイル選択ダイアログはテストハーネスで固定する。
- `PDF_ORGANIZER_INSTALLED_CORE_MULTI_PDF=1 npm run test:installed-core` は、インストール済み v0.1.4 アプリで2つのPDFを同時に取り込み、複数PDFから3セグメントを作成して3件出力する通し確認を行う。
- `npm run test:installed-output-recovery` は、インストール済み v0.1.4 アプリで出力フォルダを開く実OS導線と、部分失敗後に「失敗した1件を再出力」して成功済み行を温存したまま全件作成済みに戻る復旧導線を確認する。Windowsの対話デスクトップが必要なため、CI標準検証ではなくローカル実機スモークとして扱う。
