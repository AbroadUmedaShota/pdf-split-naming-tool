## 1. 結論

主利用を「社内Windows PC上のローカルPDF処理」とし、運用の軽さを最優先する場合、次期構成は **Python単体寄りのWindowsデスクトップアプリ** を第一候補とする。

現行の `Tauri + Next.js + Python sidecar` は、UI表現、Web技術の開発速度、自動更新には強い。一方で、Node.js、Rust/Tauri、Python sidecar の3系統を運用するため、社内向けPDF業務ツールとしては配布・保守・原因切り分けが重くなりやすい。

再設計する場合は、PDF処理、命名、状態保存を中核ドメインとして分離し、UIはそのドメインサービスを呼び出す薄い層に寄せる。

## 2. 判断前提

- 主対象はWindowsデスクトップアプリ。
- PDFはローカルファイルとして処理する。
- 複数人同時利用、クラウド保管、Web共有は現時点の主目的ではない。
- 自動更新よりも、配布・起動・保守の単純さを優先する。
- 既存実装を即破棄せず、まず技術方針を固め、必要に応じて段階移行する。

## 3. 推奨スタック

| 領域 | 推奨 | 理由 |
| --- | --- | --- |
| アプリ形態 | Windowsデスクトップアプリ | ローカルPDF処理、ファイルダイアログ、社内配布と相性がよい。 |
| UI | `PySide6` | Python単体でGUIとPDF処理を統合でき、業務画面の構築に十分な表現力がある。 |
| PDF処理 | `PyMuPDF` | 現行処理でも利用しており、ページ数取得、プレビュー、分割出力に対応できる。 |
| ドメインロジック | Python package | 命名、検証、状態保存をUIから分離し、pytestで検証しやすくする。 |
| テスト | `pytest` | 既存テスト資産と継続性がある。 |
| 配布 | `PyInstaller` one-folder | 最初は構造が見える配布形式で安定化しやすい。必要に応じて後でインストーラ化する。 |
| インストーラ候補 | Inno Setup / MSIX | one-folder運用が安定した後、社内配布要件に応じて選択する。 |

## 4. 比較

| 案 | 内容 | 良い点 | 懸念 | 評価 |
| --- | --- | --- | --- | --- |
| A. 現行改良 | `Tauri + Next.js + Python sidecar` を整理継続 | 既存資産を活かせる。UIは作りやすい。更新機能も既にある。sidecar を常駐プロセス化（v0.1.4）し、PyInstaller でビルドした exe を Tauri bundle に同梱することで配布もシンプルになった。 | Node/Rust/Pythonの保守が重い。UIプレビュー名と実出力名のズレリスクはテストで抑制済みだが、二重実装の構造的リスクは残る。 | 短期改善向き。現行継続の条件（§10）はほぼ達成済み |
| B. Pythonデスクトップ | `PySide6 + PyMuPDF + PyInstaller` | PDF処理とアプリロジックが同一言語。配布と検証が単純。 | Web UIほど表現力はない。自動更新は別途設計が必要。 | 推奨 |
| C. .NET Windowsアプリ | WinUI/WPF + PDFライブラリ | Windows業務アプリとして自然。配布・署名と相性がよい。 | PDF処理を再設計しやすい。ライブラリ選定と再実装コストが高い。 | 長期Windows標準化向き |
| D. ローカルWeb | FastAPI + React をローカル起動 | UI開発は柔軟。処理と画面を分離しやすい。 | 起動、ポート、常駐プロセス管理が面倒。業務利用者へ説明しにくい。 | 非推奨 |
| E. クラウドWeb | PDFアップロード型Webアプリ | 複数人利用や集中管理に強い。 | PDFアップロード、権限、保管、費用、監査、運用設計が必要。 | 現要件では過剰 |

## 5. 目標構造

次期構成では、復元由来の `recovery/pdf_splitter_tool/` ではなく、正式なアプリケーションパッケージとして `src/pdf_split_naming/` を置く。

```text
src/pdf_split_naming/
  domain/
    models.py
    filename_policy.py
    validation.py
  pdf/
    service.py
    preview.py
    split.py
  state/
    manager.py
    schema.py
  ui/
    app.py
    step_import.py
    step_split.py
    step_input.py
    step_output.py
  platform/
    dialogs.py
    distribution.py
    updates.py
tests/
  domain/
  pdf/
  state/
  ui_contract/
```

責務:

- `domain/`: Segment、Metadata、FilenamePolicy、Validation。UIやPDFライブラリに依存しない。
- `pdf/`: PyMuPDFを使ったページ数取得、プレビュー、分割出力。
- `state/`: `_pdf_split_state.json`、バックアップ、一時ファイル、欠損PDF時の復元方針。
- `ui/`: STEP1-4画面。業務ルールは持たず、domain/pdf/stateを呼び出す。
- `platform/`: Windowsファイルダイアログ、配布、更新確認などOS・配布に依存する処理。

## 6. 主要インターフェース

`Segment`:

- `pdf_path`
- `start_page`
- `end_page`
- `metadata`

`Metadata`:

- `box_no`
- `binder_no`
- `seq`

`FilenamePolicy`:

- テンプレート: `{box_no:0>2}_{binder_no:0>2}_{seq:0>3}.pdf`
- 必須入力チェック。
- Windowsファイル名正規化。

`State`:

- 入力PDF一覧。
- 出力フォルダ。
- 分割セグメント。
- セグメントごとの箱No、バインダーNo、連番。
- 現在PDF。
- 現在ページ。

`PdfService`:

- `page_count(pdf_path)`
- `preview(pdf_path, page_no)`
- `split(segment, output_path)`

`ExportService`:

- `preflight(output_dir, segments)`
- `export(output_dir, segments)`
- 同一出力リクエスト内の一意名採番。
- 既存同名ファイルの検出と上書き防止。

## 7. 設計原則

- 命名仕様、ファイル名正規化、同一出力リクエスト内の一意名採番、既存同名ブロック、状態保存はUIに置かない。
- UIはSTEP1-4の表示・入力・遷移に集中する。
- PDF処理はPyMuPDF依存を `pdf/` に閉じ込める。
- 1始まりページ番号をアプリ内の業務表現とし、ライブラリ境界でのみ0始まりへ変換する。
- 状態保存は一時ファイル、バックアップ、破損退避を標準動作にする。
- 配布はまずone-folderで安定化し、利用実績を見てインストーラ化する。

## 8. 移行方針

段階移行とし、現行アプリをすぐに破棄しない。

1. 技術方針の確定。
2. `domain/` 相当の命名、検証、Segment、Stateを現行Python処理から抽出する。
3. `pdf/` サービスをPyMuPDFの薄いラッパーとして整える。
4. `state/` を正式スキーマ化し、欠損PDF時の復元方針を決める。
5. PySide6でSTEP1-4の最小UIを試作する。
6. 現行Tauri版と同じ現場PDFで、取込→分割→入力→出力を比較する。
7. 配布方式をPyInstaller one-folderで検証する。
8. 問題が少なければ新構成へ主軸を移し、必要に応じてInno Setup / MSIX / 更新確認を追加する。

移行中の運用制約:

- 現行Tauri版のGitHub Releases / updater配布ラインと、次期PySide6 / PyInstaller配布ラインは同一の更新経路として扱わない。
- 次期構成を試作・検証する間は、現行Tauri版を安定版として残し、利用者向け配布物の名称、配置、バージョン表記を分ける。
- `_pdf_split_state.json` は既存利用者の作業状態を保持するファイルであるため、次期構成は読み込み互換を維持する。互換を壊す場合は、移行ツールまたはバックアップ付き変換を必須にする。
- ロールバック時に旧版が読めない状態ファイルを残さない。状態スキーマを変更する場合は、旧形式を温存するか、明示的なスキーマバージョンを持たせる。
- 次期構成の配布判断は、現場PDFでの通し確認に加え、既存状態ファイルの読み込み、現行版への戻し手順、配布先の混在リスクを確認してから行う。

## 9. テスト方針

domain:

- 命名テンプレート。
- ゼロ埋め。
- 必須入力。
- Windowsファイル名正規化。
- 同一出力リクエスト内の `_2`, `_3` 一意名採番。
- 出力先に既存同名ファイルがある場合の `output_exists` ブロック。

pdf:

- ページ数取得。
- 1ページ分割。
- 任意範囲分割。
- 1始まりページ番号の扱い。

state:

- 保存。
- 復元。
- bak復元。
- tmp復元。
- 欠損PDF時の扱い。

ui:

- STEP1→STEP4 の通し操作。
- 1024px以上で主要操作が見切れないこと。
- 未入力、出力先未指定、既存同名ファイルなどの警告表示。

配布:

- PyInstaller成果物で起動できる。
- 配布成果物からPDF取込、分割、入力、出力まで確認できる。

## 10. 現行構成を継続する場合の条件

次の条件を満たす場合は、短期的には現行 `Tauri + Next.js + Python sidecar` を継続してよい。

- 既存のTauri更新機能を活かす必要がある。
- Web UIの表現力を優先したい。
- Node/Rust/Pythonのビルド・配布・問い合わせ対応を継続できる体制がある。
- UIプレビュー名とsidecar実出力名、同一出力リクエスト内の一意名採番、既存同名ブロック、状態保存などの二重実装リスクをテストで抑えられる。

**現状（v0.1.4）との照合**: 上記条件はいずれも達成済み。Tauri updaterは稼働中、UIプレビュー名と実出力名の一致は回帰テストで固定済み、sidecarはPyInstallerでビルドした exe をTauri bundleに同梱する形で配布を簡略化済み（`docs/05_実装状況.md` §3「sidecar配布」参照）。PySide6移行の判断は `docs/04_移行ロードマップ.md` に従う。

この場合でも、命名、検証、同一出力リクエスト内の一意名採番、既存同名ブロック、状態保存はPython側の単一責務へ寄せる。

## 11. 判断保留事項

- 自動更新を次期Python構成でも必須にするか。
- one-folder配布で足りるか、インストーラが必要か。
- PySide6 UIで現行の業務効率を満たせるか。
- 旧アプリ由来の機能をどの段階で再評価するか。
- 現行Tauri版をいつ保守停止にするか。
- 現行Tauri版と次期Python版の並行配布期間をどれくらい確保するか。
- `_pdf_split_state.json` の互換範囲とスキーマバージョン方針をどう定義するか。

## 12. 現時点の推奨

短期:

- 現行アプリは維持する。
- 本書を技術判断の基準にする。
- 命名、正規化、同一出力リクエスト内の一意名採番、既存同名ブロック、状態保存の単一責務化を優先して検証する。

中期:

- Python package + PySide6 の試作を作り、現場PDFで通し確認する。
- 配布はPyInstaller one-folderから始める。

長期:

- PySide6構成で運用が軽くなることを確認できたら、次期主構成へ移行する。
- 自動更新やインストーラは、実運用の配布課題が明確になってから追加する。
