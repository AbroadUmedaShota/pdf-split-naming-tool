1. テスト目的

PDFを取り込み、ページ単位で分割し、箱No・バインダーNo・連番（＋任意の追加項目）で命名して、既存ファイルを上書きせずに分割PDFを出力するという中核業務（STEP1取込 → STEP2分割 → STEP3入力 → STEP4出力）が、ヨシダエルシス向けの基本運用で迷わず・データを壊さず・正しい名前で完了できることを確認する。

このテストで得たい確信は次の3点に集約する。

  • 中核動線の動作確認: 取込から出力までを通しで実行でき、画面に表示した出力名と実際に生成されるファイル名が一致する。直近で「プレビューが全く表示されない（PNG/JPEG不一致）」という中核経路のバグが発覚しており、中核動線が長く未テストだった事実をリスクに反映する。
  • データ破壊の不在確認: 既存PDFの上書きが起きない、状態保存・復元で入力値が無言で失われない・古い値が復活しない、分割点変更でメタデータが壊れない。
  • リグレッション境界の明確化: 自動テスト（pytest／nodeロジック検査／Rust単体）で守れる範囲と、sidecar実体・実機Tauri・アプリ内updaterのように人間の手動受入でしか確定できない範囲を切り分け、リリース判断者が「何が自動で緑か」「何が人手で未確認か」を区別できるようにする。

本計画は確信を与える範囲と与えられない範囲を明示することを重視する。E2E自動テストは現状ゼロであり、Playwrightで自動化できるのは Tauri invoke をモック化したUI・状態・ナビゲーション検証までである。実PDFの分割・プレビュー・出力など sidecar 実体を伴う検証は人間実行（手動受入）として扱う。

3. テストアイテム

3.1 機能・画面

  • STEP1 取込: PDF複数選択／一覧から除外／全クリア、出力フォルダ指定/リセット、共通値（箱No・バインダーNo）、状態保存。次へ進む条件（PDF1件以上＋出力先指定）。
  • STEP2 分割: ページプレビュー、ページ状態一覧、前後ページ・ページ番号直接入力・複数PDF間移動、ズーム/幅合わせ/全体表示、現在ページ前で分割・最後の分割取り消し・選択分割削除・現在PDF全解除、検索（現在PDF固定）・検索語ハイライト・OCR本文（テキスト層）表示・候補検索・白紙候補。
  • STEP3 入力: セグメント選択、箱No・バインダーNo・連番（必須3項目）、追加項目最大2件（prefix/suffix）、OCR本文からの追加項目転記、連番再採番、出力名プレビュー。
  • STEP4 出力: 出力前チェック（preflight）、出力実行（export）、出力予定一覧、チェック結果、出力ログ、既存ファイル衝突時の一意名採番（_2/_3）・上書き防止。
  • 更新/配布: Tauri updater による更新検出・インストール・再起動後バージョン表示。

3.2 sidecarコマンド（API境界）

pdf_info / page_preview / page_thumbnail / page_text / search_text / search_highlights / index_candidates / blank_candidates / preflight / export / state_load / state_save。常駐 serve モード（JSON Lines）と oneshot 退避モード（PDF_ORGANIZER_SIDECAR_MODE=oneshot）の両系統。

3.3 命名・状態の中核ロジック

  • 命名テンプレート {box_no:0>2}_{binder_no:0>2}_{seq:0>3}.pdf と追加項目 affix_defs（最大2件・prefix/suffix・空値除外・後方互換）。
  • フロント（filename-policy.ts）と Python（domain）のファイル名ビルダ一致（strip・Windows禁止/予約名正規化を含む）。
  • 状態保存 _pdf_split_state.json（＋ .bak.json / .tmp）。主要フィールド、スキーマ検証、tmp→state コミット、bak退避、破損退避・復旧、pid付きtmp、残骸掃除。

3.4 既存テスト資産（正式回帰資産）

  • pytest 19ファイル（test_sidecar / test_state_recovery_failures / test_output_name_parity / test_iss030_nf ほか）。
  • nodeロジック検査（check-all.mjs が集約する6検査：filename-policy / segment-state / restore-state / output-state / step-render / preview-cache）。
  • Rust 単体テスト（apps/desktop/src-tauri/src/lib.rs。常駐サイドカー管理ほか。本数は 要確認／ISS-014 記載では常駐分13件）。
  • 集約検証ルート scripts/verify.ps1（pytest ＋ desktop JS回帰 ＋ typecheck）、scripts/sample-pdf-smoke.ps1（サンプルPDFのsidecarスモーク）。

4. テストスコープ

4.1 In Scope（自動・本フローで整備）

  • sidecarコマンドの正常系・異常系・境界（ページ範囲外・空入力・型不正・エラー応答形状）— pytest を主軸に拡充。
  • 命名ロジックと FE/BE 一致（空白混入・禁止/予約文字・affix の prefix/suffix・空値除外・三者正規化一致 NF-D1）— pytest（output_name_parity）＋ nodeロジック検査（filename-policy）。
  • 状態保存・復元のロジックと耐障害（部分復元・欠損列挙・current_page クランプ・破損退避・tmp/bak 復旧・pid衝突・残骸昇格防止）— pytest（state_recovery / state_recovery_failures / state_compat）。
  • preflight/export の判定ロジック（出力先未指定・セグメントなし・既存衝突一意名・部分失敗 summary・preflight_blocked）— pytest。
  • page.tsx 由来の純ロジック（segmentKey / buildSegments / ステップ判定 / プレビューキャッシュ / 復元 / 出力状態）— nodeロジック検査。
  • 新規 E2E（Playwright・Tauri invoke モック）: UI状態・4工程ナビゲーション・ステッパー誘導・busy/二重実行ガードの可視状態・確認ダイアログ発火・トーン表示・部分失敗の attention 表示・ハイライト残留（NF-U5）のUI挙動。invoke はモック応答で駆動する範囲に限定。

5. テスト終了条件

5.1 自動テスト

  • pytest（recovery/tests/）が全件パス。本フローで追加するケースを含む。
  • nodeロジック検査（npm run test:desktop = check-all、および個別 test:* スクリプト）が全件パス。
  • 型チェック npm run typecheck（tsc --noEmit）が成功、next build が成功。
  • 集約検証 scripts/verify.ps1 が成功。
  • 新規 Playwright E2E（invokeモック範囲）が全件パス、または未実装ケースが 未実装テストケース_E2E自動.md に可視化されている。

5.2 手動受入（人間実行）

  • 第4.2節の Out of Scope 各項目について、手動受入テストケース（人間実行）が用意され、実行結果（Pass/Fail/未実施）が最終レポートに可視化されている。未実施は「未実施」として明示し、緑扱いにしない。
  • 中核動線（取込→分割→入力→出力の通し）が実PDFで1回以上成功し、画面の出力名と実ファイル名が一致することを人間が確認。
  • 実機確認チェックリスト（issues.md 記載の12項目）の各項目が Pass / Fail / 未実施で記録される。
  • アプリ内 updater の一気通し（更新検出→インストール→再起動後バージョン表示）が Pass / Fail / 未実施で記録される。

5.3 欠陥・残リスク・成果物品質ゲート

  • Fail として記録された不具合は、起票（BUGリスト）され、リリース判断者がリスク受容 or 修正済みを判断できる状態にある。
  • 高リスク項目（中核動線・プレビュー表示・データ破壊・updater）で未実施または Fail がある場合は、リリース可否判断にその残リスクが明記される（サイレントに緑化しない）。
  • 後続QA成果物（テスト計画・分析・設計・テストケース）のレビューで、未解決の P0/P1 指摘が残っていない。残る場合は明示的に承認または DEFERRED として記録されている。
  • OPEN/DEFERRED 既知課題（ISS-027/028 ほか）について、リリース時点の扱い（修正/受容/次タスク）が判断材料として残されている。
  • cargo test（ビルド機）の実施有無が記録され、未実施なら残リスクとして可視化されている。

組織的なサインオフ手順・承認者・しきい値の数値基準は本計画の参照資料に定義がないため、要確認（第11節）とする。
