1. 設計対象

テスト分析の観点 TV001-TV075（75観点）と Source Structure Inventory SSI001-SSI020、テスト計画のリスク R001-R014・アプローチ TA001-TA015・ユースケース UC1-UC10 を入力とする。各観点を設計パターンへ展開し、消費すべき有限集合・境界（命名禁止文字9種・予約名・affix 0-3件・seq 桁数1-9・ページ範囲端・検索上限200・白紙閾値0.985・8秒予算）を設計に可視化する。

1.1 実行レーン（設計上のケース種別の振り分け基準）

各 TD は1つの主たる実行レーンを持つ。レーンは以下の3区分。後続のテストケースはこのレーン区分に従って テストケース_コードベース.md / テストケース_E2E自動.md / テストケース_人間実行.md へ振り分ける。

  • レーン①自動コードベース: pytest（recovery/tests/）・node check-*.mjs（apps/desktop/scripts/）・Rust 単体（cargo・実行可否は Q2）。sidecar 純ロジック・命名パリティ・preflight/export 判定・状態往復/耐障害・page.tsx 純ロジック。テストレベル/タイプ は Unit / Integration / Regression。
  • レーン②自動E2E: Playwright（Tauri invoke をモック応答で駆動する範囲）。UI 状態・4工程ナビ・ステッパー誘導・busy/二重実行ガードの可視状態・確認ダイアログ発火・トーン/attention 表示・ハイライト残留などの UI 挙動。実 sidecar・実 PDF レンダリングは扱わない。テストレベル/タイプ は E2E。**Playwright 導入有無は未確定（質問票 DQ03＝分析 Q3）。** E2E レーンの TD は導入前提が確定するまで 状態=質問待ち とし、未導入の場合は同等観点を手動受入（探索的 UI 確認）へ退避できるよう判定観点を記述する。
  • レーン③人間実行（手動受入）: 実 PDF の実描画・実分割・実出力・既存衝突回避の実フォルダ確認・実機 Tauri 12項目・updater 一気通し・幅確認・スキャン/日本語パス PDF。sidecar 実体・PyMuPDF 依存・OS ダイアログ・実機ビルド依存のため自動化不可。テストレベル/タイプ は Manual。未実施を緑化しない判定観点を記述する。

1.2 期待結果が未確定の扱い

参照資料で合否判定点が確定できないもの（性能サインオフ基準 Q1／stale 復活のセッション内仕様 Q9／白紙閾値境界の正式許容度 Q10／分割マーカー HTML 構造の合否範囲 Q11／cargo 実行環境・本数 Q2／Playwright 導入 Q3／手動受入実施体制 Q4／実テストデータ Q5／OPEN課題扱い Q6／updater 旧版配布物 Q7／DEFERRED スコープ Q8）は推測で固定せず、該当 TD を 状態=質問待ち・該当フィールドを 要確認 とし、## 4. 質問待ち項目 と設計質問票へ相互リンクする。
