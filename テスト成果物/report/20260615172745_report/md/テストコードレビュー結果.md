| 優先度 | 観点 | 問題 | 場所 | 影響 | 修正方針 |
|---|---|---|---|---|---|
| P2 | アサーション強度／決定性 | TC-CB-132（blank_candidates 部分結果）が `time_budget=0.0` で `partial`/`scanned_until` の型・非負のみを検証し、TD035 の partial 意味づけを弱く検証していた | test_tc_cb_132 | budget 超過時の途中打ち切り挙動が実質未検証 | 既存 `test_blank_candidates_partial_result_when_budget_exceeded`（test_iss030_nf.py）と同じ条件分岐パターンに合わせ、partial 時は `scanned_until < total` かつ候補が走査範囲内、非partial 時は `scanned_until == total` を検証するよう強化 |
| P2 | 実対象検証／トレーサビリティ正確性 | TC-CB-095 のテスト名・docstring が「stale segment_metadata フィルタ」を検証すると示唆していたが、その挙動は Python sidecar/state_schema に存在せず（フロント segment-state.ts の reconcileSegmentMetadataForPdf＝ISS-003 が担当。node check-segment-state.mjs＝TC-CB-106/107 でカバー）、実テストは正当値の往復保全を検証していた（ラベル不一致） | test_tc_cb_095 | テストの主張と実検証内容の乖離（誤解を招く） | テスト名を `..._segment_metadata_roundtrips_intact` に改め、docstring に「stale フィルタは node 層が担保、Python は往復保全のみ保証」と明記。検証内容（往復保全）は妥当なため維持 |

判定理由（priority 根拠）:

- 上記2件はいずれも実行は通る（runnable）が、依拠前に直すべき項目のため fix-worthy。TC-CB-132 は高リスク挙動の弱アサーション、TC-CB-095 は対象/ラベル不一致で、いずれも本スキルの P2 基準に該当。
- P0/P1 は検出されなかった: テストは実 API（handle_request / build_yoshida_filename_preview / normalize_state_payload / StateManager / PdfService.search_text・blank_candidates）を直接駆動しコピーロジックを検証していない。質問待ち（TC-CB-146〜151）を不正なアサーションへ変換した箇所はなく、未実装一覧へ正しく退避。export/state は実ファイルシステムで検証し、失敗注入は scoped monkeypatch（Path.replace/write_text・PdfProcessor.split_pdf）で行い後始末も pytest 管理下。
