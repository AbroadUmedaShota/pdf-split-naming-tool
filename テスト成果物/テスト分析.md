# テスト分析

対象プロダクト: PDF分割命名ツール（apps/desktop の Tauri + Next.js デスクトップアプリ ＋ recovery/pdf_splitter_tool の Python sidecar）
作成日: セッション内作成
位置づけ: テスト計画書（`テスト成果物/テスト計画書.md`）のテストアプローチ TA001-TA015 を観点レベルへ分解する。詳細手順・期待値は記載せず、後続のテスト設計・テストケースが分解できる粒度にとどめる。
正本仕様: `docs/01_要件定義書.md`

## 1. 分析対象

計画書のリスク R001-R014、テストアプローチ TA001-TA015、ユースケース UC1-UC10 を入力とし、各 TA に対応するテスト観点（TVxxx）を導出する。中核動線（取込→分割→入力→出力）の通し動作・命名一致・データ破壊不在・リグレッション境界を厚く扱う。E2E は現状ゼロで、Playwright で自動化できるのは Tauri invoke をモック化した UI・状態・ナビゲーションの検証範囲に限定する。sidecar 実体を伴う検証（実 PDF のプレビュー描画・分割・出力）・実機 Tauri・アプリ内 updater は人間実行（手動受入）として観点を整理する。

三パスで導出した。

- Pass 1（仕様準拠）: 要件定義書・命名仕様・状態保存仕様・sidecar コマンド表・既知課題 ISS-001〜030・既存テスト資産から観点を導出。
- Pass 2（汎用QA）: 対象タイプ（デスクトップアプリ UI ＋ CLI/IPC 境界の sidecar ＋ ファイル出力 ＋ 永続化 JSON ＋ 自動更新）に一般的に必要な QA 観点で Pass 1 の不足を補完。
- Pass 3（追加観点検討）: バグ逃し想定・高リスク組合せ・下流設計準備性・トレーサビリティ欠落の再点検で残りの欠落観点を追加。
